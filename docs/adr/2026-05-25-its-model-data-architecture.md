# ADR: ITSモデルと正本データの対応方針

## Status

Accepted

## Date

2026-05-25

## Context

探Qメイトでは、ITS（Intelligent Tutoring System）の内部構成として、学習者モデル、教授モデル、ドメイン/課題モデル、評価/観測モデルを扱う。

一方で、これらのモデル名は教育工学上の概念であり、必ずしもそのまま永続DBテーブルの正本を意味しない。特に学習者モデルについては、「ユーザーごとに形成されるモデル」をどこまで永続化するか、またテーマ・問い・仮説などの探究データを学習者モデルに含めるべきかが曖昧になりやすい。

現行実装では、ITSモデルはチャット応答時に内部スナップショットとして構築され、長期的なユーザー別傾向は `its_observation_profiles` の集約型観測プロファイルを通じて参照される。したがって、モデル概念、正本データ、更新タイミング、紐づけ過程を明確にする必要がある。

## Decision

探QメイトのITSモデルは、MVP時点ではスナップショット型として扱う。

学習者モデル本体をユーザーごとの永続正本テーブルとして新設しない。ユーザーごとの長期的な個別最適化は、日誌由来の観測レコードと集約型観測プロファイルを通じて行う。個別最適化の精度が不足した場合は、永続型学習者モデルを追加する前に、集約型観測プロファイルの内容、形成過程、参照方法を見直す。

テーマ、問い、仮説、探究フェーズ、成果物などは学習者モデルの正本ではなく、ドメイン/課題モデルの正本データとして扱う。学習者モデルは、それらを「現在の関心」「支援の文脈」「困り感の背景」として参照する。

## ITSモデルの位置づけ

### 学習者モデル

学習者モデルは、現在の生徒に合わせて支援を調整するための内部状態スナップショットである。

主な内容:

- 学年帯
- 学校・クラスなどの制約
- 現在の興味・関心
- 自分ごと感のサイン
- 困り感・迷いのサイン
- 支援の受け取り方
- 選択・保留・拒否のサイン
- 直近感情
- 観測プロファイル要約

形成タイミング:

- `/chat` の応答生成前に、`build_its_context()` で都度構築する。
- 正本として永続化せず、チャットターンログには当該ターンの内部スナップショットとして保存する。

更新タイミング:

- 学習者モデル自体は更新しない。
- 材料となるプロフィール、探究データ、会話履歴、観測プロファイルが更新されることで、次回構築されるスナップショットが変わる。

### 教授モデル

教授モデルは、探Qメイトがどのように支援するかを決める内部制御モデルである。

主な内容:

- 応答スタイル
- 質問予算
- 開示レベル上限
- 行動促進の強さ
- 優先支援タイプ
- 避けるべきパターン
- 方針メモ

形成タイミング:

- `/chat` の応答生成前に、学習者モデルとドメイン/課題モデルを参照して都度構築する。
- `TutorOrchestrator` が教授モデルを参照し、支援タイプ、発話アクト、開示レベル、質問予算、行動促進の強さを決める。

更新タイミング:

- 教授モデル自体は、開発者が定義する支援方針と実装ロジックにより更新される。
- ユーザーごとの永続データとしては保存しない。
- 各ターンで選ばれた教授方略は `its_chat_turn_logs` に保存する。

### ドメイン/課題モデル

ドメイン/課題モデルは、探究課題そのものの現在地を表す内部スナップショットである。

主な内容:

- 探究フェーズ
- テーマ
- 問い
- 仮説
- 成果物
- 現在話題
- 品質シグナル

形成タイミング:

- `/chat` の応答生成前に、プロフィール、既存プロジェクト文脈、生徒発話、会話文脈から都度構築する。

更新タイミング:

- ドメイン/課題モデル自体は正本ではない。
- テーマ、問い、仮説などの正本データが更新されることで、次回構築されるスナップショットが変わる。

### 評価/観測モデル

評価/観測モデルは、生徒を評価するためではなく、支援過程とAIの観測仮説を記録し、次回以降の支援調整に使う内部モデルである。

主な内容:

- 集約型観測プロファイル要約
- 直近の観測レコード
- 応答リアクションシグナル
- 根拠ソース
- 注意書き

形成タイミング:

- `/chat` の応答生成前に、`its_observation_profiles` を参照して都度構築する。
- 日誌生成時には、AI日誌下書きから観測レコードを作成する。

更新タイミング:

- 日誌生成時に `its_observation_records` が追加される。
- 観測レコード追加後、直近レコードを集約して `its_observation_profiles` が insert または update される。
- チャットターンごとの支援プロセスは `its_chat_turn_logs` に保存される。

## 正本データとの対応

| データ | 正本 | ITS内の扱い |
|---|---|---|
| 生徒ID | Supabase Auth user ID / legacy user ID mapping | 全ITSログと観測プロファイルの紐づけキー |
| 学年 | `profiles.grade` | 学習者モデルの `grade_band` へ変換 |
| 学校・クラス | `profiles.school_id`, `profiles.class_name` など | 学習者モデルの `school_constraints` |
| テーマ | `profiles.theme` / legacy project context | ドメイン/課題モデルの `theme`、学習者モデルの `interests` の材料 |
| 問い | `profiles.question` / legacy project context | ドメイン/課題モデルの `inquiry_question` |
| 仮説 | `profiles.hypothesis` / legacy project context | ドメイン/課題モデルの `hypothesis` |
| 探究フェーズ | profile/projectの進捗、または発話からの推定 | ドメイン/課題モデルの `phase` |
| 現在発話 | chat request body | 学習者モデル、ドメイン/課題モデル、教授方略選択の最優先材料 |
| 会話履歴 | chat logs | 文脈構築と観測モデルの根拠ソース |
| AI日誌下書き | diary draft generation result | `its_observation_records.observation` の材料 |
| 集約型観測プロファイル | `its_observation_profiles` | 学習者モデルと観測モデルが参照する長期傾向 |
| 支援方略 | `TutorOrchestrator` の決定 | `its_chat_turn_logs` にターン単位で保存 |

## 形成・紐づけフロー

### チャット応答

```text
chat request
  ↓
conversation_id の取得または作成
  ↓
プロフィール、既存プロジェクト文脈、会話履歴を取得
  ↓
student_user_id で its_observation_profiles を取得
  ↓
build_its_context()
  ↓
学習者モデル、教授モデル、ドメイン/課題モデル、評価/観測モデルのスナップショットを構築
  ↓
TutorOrchestrator が支援方略を選択
  ↓
AI応答生成
  ↓
通常のチャットログ保存
  ↓
its_chat_turn_logs にITSスナップショット、支援方略、教育的検証結果を保存
```

### 日誌・観測プロファイル

```text
diary draft generation
  ↓
対象日の会話、前回日誌、感情コンテキスト、集約型観測プロファイルを取得
  ↓
AI日誌下書きを生成
  ↓
AI日誌下書き、参照文脈、観測シグナルを its_observation_records に保存
  ↓
student_user_id の直近観測レコードを集約
  ↓
its_observation_profiles を insert または update
  ↓
次回以降の chat request で学習者モデルと観測モデルが参照
```

## アーキテクチャ上の境界

- ITSモデルは内部制御用であり、生徒向けチャットAPIや先生向けAPIにそのまま返さない。
- `its_chat_turn_logs` は raw conversation log を保存する場所ではない。保存対象はモデルスナップショット、支援方略、教育的検証結果である。
- `its_observation_profiles` はAIの見立ての集約であり、生徒の事実や能力評価として扱わない。
- 現在発話と過去観測が矛盾する場合、現在発話を優先する。
- テーマ、問い、仮説はドメイン/課題データの正本を参照する。学習者モデルへ重複正本化しない。
- 個別最適化の改善は、まず集約型観測プロファイルの情報設計、抽出ロジック、集約件数、古い観測の扱いを見直す。

## Consequences

### Positive

- 学習者モデルが固定的な生徒ラベルになりにくい。
- テーマ、問い、仮説の正本がドメイン/課題データ側に残り、責務が明確になる。
- チャットごとの支援判断を追跡できる。
- 個別最適化の改善対象が `its_observation_profiles` に集約され、MVPの実装複雑度を抑えられる。

### Negative

- 長期的な個別最適化は、集約型観測プロファイルの質に依存する。
- 学習者モデルを単一レコードとして直接閲覧・編集する運用には向かない。
- 観測プロファイルの集約ルールが粗い場合、支援適応の精度が頭打ちになる可能性がある。

## Future Considerations

- 集約型観測プロファイルに含めるべき支援適応シグナルの見直し
- 古い観測の減衰ルール
- 生徒がAIの見立てに違和感や修正を示せるUI
- 先生向けに観測情報を出す場合の要約粒度と権限境界
- 永続型の支援適応プロファイルを導入するかどうか

永続型を検討する場合も、テーマ、問い、仮説を学習者モデルへ移すのではなく、支援の受け取り方、困りやすい場面、好む足場などの支援適応情報に限定して設計する。
