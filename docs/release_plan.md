Phase 1：プロダクトのリリース品質を作る

期間目安は 1〜2週間です。

ここでは新機能を増やすより、一本の体験を通すことに集中します。

1. オンボーディング / プロフィール

やることはシンプルです。

必須項目を最小化
任意項目はスキップ可能
離脱しても戻れる
保存失敗時に再試行できる
最後に「次に何をすればいいか」が分かる
2. 日誌生成 v1

ここは完成版を目指さず、次の最低ラインで十分です。

1つの代表入力導線
生成中表示
生成失敗時の再試行
結果の編集
手動入力へのフォールバック
保存
3. 共有サマリー v0

これは学校実証にも直結するので、かなり重要です。

生のAI送信データは学校側から見えない
学校に共有されるのは要約・整理済み情報のみ
共有前に生徒が確認する
共有済み / 未共有が本人に分かる
先生側は「今見えているのが raw ではなく summary である」ことが分かる表示にする
4. 品質と観測

最低限これを入れます。

主要画面の empty / loading / error
問い合わせ/不具合報告
重要イベント計測
主要エラーのログ
手動で確認できる簡易管理メモ
5. リリース判定指標

最初の仮目標としてはこのくらいで十分です。

Phase 3：先行公開βの運用開始

期間目安は 2〜4週間です。

ここでの目的は「人数を増やす」ではなく、濃い学びを得ることです。

個人向け先行βの最初の設計

おすすめは、最初の cohort を分けることです。

Cohort A
18歳以上の教育関心層、大学生、教育事業の人、教員志望など
目的は UX と言語の検証

Cohort B
保護者同意済みの高校生、または学校経由の高校生
目的は実際の探究・振り返り文脈での検証

Khanmigo も、18歳未満の利用は保護者か学校パートナー経由に寄せています。探Qメイトも、未成年の直接募集は段階的にした方が安全です。

先行βの運用リズム
週1の改善ログ公開
週1の短いアンケート
1〜2週間に1回の少人数共有会
Discord / Slack / LINEオープンチャットのどれか1つ
重要な要望はロードマップ投票に反映

Dia は学生向けに academic email での早期アクセスや Discord 参加導線を置いていて、MagicSchool も educator community と Pioneer program を持っています。探Qメイトも「ただのモニター」ではなく、初期メンバー / ラボメンバーとして運用した方が熱量が上がります。

学校現場での実験計画

学校現場では、最初から「導入」ではなく、6週間前後の実証として回すのがよいです。

実証の設計

各校で、最初はこれくらいが適切です。

1校あたり 1人の champion teacher
1クラスまたは1学年の小さめ単位
6週間
週1回以上の活用

OpenAI の rollout / launch guide でも、小さな task force、baseline survey、use-case discovery、champions network を置くことが重視されています。学校でもまったく同じです。

4. 安心して使える理由

ここは探Qメイトの独自性になります。

学校に共有されるのは、生データではなく整理済みの共有情報
共有前に本人が確認する
raw data は学校から見えない
先行公開βとして改善中だが、扱いは慎重にしている

MagicSchool、Brisk、SchoolAI、Notion も、価値だけでなく privacy / safety / controls / visibility を前に出しています。探Qメイトもここを逃さない方がいいです。
