# 探Qメイト Webアプリ公開前チェックレポート

作成日: 2026-04-13

参照基準:
- [Webサービス公開前のチェックリスト](https://zenn.dev/catnose99/articles/547cbf57e5ad28)

## 概要

探Qメイトの現状実装を、公開前チェックの観点で静的レビューした結果を整理する。

今回の確認では、公開を止めるべき重大項目として以下を確認した。

1. 先生向け日誌 API に権限制御不足がある
2. 認証トークンが `localStorage` に保存されている
3. サーバー内部例外の詳細がクライアントへ露出している
4. CORS と公開用オリジン設定が本番向けに整理されていない
5. OGP / canonical / favicon などの公開メタデータが不足している

あわせて、現ワークスペースではフロントエンドのビルドとバックエンドのテストが再現できず、公開前の最終確認環境として不十分だった。

## 課題・問題点

### 1. 先生向け日誌 API の権限制御不足

対象:
- `apps/backend/routers/diary_router.py:175-210`
- `apps/backend/services/diary_service.py:260-352`

内容:
- 先生向け API に `TODO` のまま権限チェック未実装
- `get_teacher_dashboard` は提出済み日誌を広く取得する構造
- `get_student_diaries` は指定された `student_id` に対してそのまま取得する構造
- サービス層でも teacher と student の関係性を検証していない

リスク:
- 認証済みユーザーが他生徒の日誌、氏名、メールアドレスにアクセスできる可能性がある
- 個人情報漏えいにつながる
- 公開前に必ず塞ぐべき P0

### 2. 認証トークンの `localStorage` 保存

対象:
- `apps/frontend/src/lib/supabase.ts:17-24`
- `apps/frontend/src/utils/tokenManager.ts:47-93`

内容:
- Supabase セッション保存先が `window.localStorage`
- さらに独自の `auth_tokens` に access token / refresh token を複製保存している

リスク:
- XSS が成立した場合にトークンを読み取られる
- 記事の公開前チェックで挙がっている Cookie 保護方針と比べて防御力が低い
- 認証情報保護の観点で公開前に見直したい

### 3. サーバー例外詳細のクライアント露出

対象:
- `apps/backend/routers/diary_router.py:99-144`

内容:
- 500 エラー時に `str(e)` をそのまま `detail` に入れて返している
- フロント側は返却された `detail` をそのまま表示する作り

リスク:
- 内部エラー内容、依存サービス失敗、実装詳細、クエリ失敗要因などが露出する
- 攻撃や調査の材料を利用者に渡すことになる
- 公開前チェックの「サーバーで発生したエラーメッセージをそのままブラウザに表示しない」に抵触

### 4. CORS / 公開オリジン設定の未整理

対象:
- `apps/backend/main.py:81-93`
- `apps/frontend/nginx.conf:55-69`

内容:
- バックエンドの CORS 許可元が `localhost` と `dev.tanqmates.local.test` 中心
- フロント nginx は `https://demo.tanqmates.org` を前提に CORS ヘッダーを固定
- 環境変数化されておらず、開発・デモ・本番の切り替えが設定ファイル依存

リスク:
- 本番公開時に API 通信がオリジン不一致で失敗する可能性
- 意図しない緩い設定または不足設定を残しやすい
- デプロイ再現性が低い

### 5. 公開メタデータ・アイコン不足

対象:
- `apps/frontend/index.html:5-11`
- `apps/frontend/vite.config.ts:19-41`
- `apps/frontend/public`

内容:
- `index.html` に title / description / `lang="ja"` はある
- しかし canonical、OGP、Twitter Card が未設定
- ルートアイコンが `/vite.svg` のまま
- PWA で参照している `favicon.ico`、`apple-touch-icon.png`、`masked-icon.svg`、`pwa-192x192.png`、`pwa-512x512.png` が `public` 配下に見当たらない

リスク:
- SNS シェア時の見え方が崩れる
- favicon や端末ホーム追加時の体験が不完全
- 公開物としての仕上がり不足

### 6. 404 / エラーページ要件の未整備

対象:
- `apps/frontend/src/App.tsx:281-282`

内容:
- 未定義ルートをすべて `/` にリダイレクトしている
- 専用の 404 ページがない

リスク:
- エラー系ページが 40x にならない
- noindex 制御もできない
- ユーザーと検索エンジン双方に不親切

### 7. XSS 面で注意が必要な描画

対象:
- `apps/frontend/src/components/Diary/DiaryEditor.tsx:498-503`

内容:
- `contentEditable` と `dangerouslySetInnerHTML` を組み合わせている
- 現時点では下書き用途だが、可変文字列を HTML として描画する構造

リスク:
- 将来入力元が広がると XSS の温床になりやすい
- トークンを `localStorage` に置いている現状と組み合わさると影響が大きい

### 8. ビルド・テスト再現性不足

確認結果:
- フロントエンドビルドは `vite` 実行不可
- `node node_modules/vite/bin/vite.js build` でも `esbuild` 欠落
- バックエンドは `.venv` 内に `pytest` 未導入

リスク:
- 「公開前にビルド成功・テスト成功を確認した」という状態を現環境で証明できない
- 依存関係やセットアップ手順の揺れが残っている

## 改善方針

### 最優先で修正する項目

1. 先生向け日誌 API に厳密な認可を入れる
- teacher ロールを必須化する
- teacher が閲覧可能な student の集合を明示的に絞る
- `teacher_id` と `student_id` の関係を DB 上で検証する
- ルーターだけでなくサービス層でも不正取得を拒否する

2. 500 エラーの返却内容を利用者向けメッセージへ置き換える
- サーバー内部ログには詳細を残す
- API レスポンスは一般化した文言にする
- 監視基盤で詳細を拾える状態にする

3. 認証トークンの保存方針を見直す
- 少なくとも独自 `auth_tokens` の二重保存はやめる
- 長期的には httpOnly Cookie ベースを優先検討する
- もし Supabase の制約で `localStorage` 運用を続けるなら、XSS 対策を厳格化した上でリスク受容を明文化する

4. 本番 CORS とデプロイ設定を環境変数化する
- 許可オリジンを環境別に切り替える
- フロント nginx とバックエンド CORS の定義元を揃える
- `demo` / `dev` / `prod` が混在しないようにする

### 公開前に揃える項目

5. メタデータと公開資産を追加する
- favicon
- apple-touch-icon
- OGP画像
- `og:title`
- `og:description`
- `og:url`
- `og:image`
- `twitter:card`
- canonical URL

6. 404 / 500 表示導線を作る
- フロントで Not Found ページを用意する
- 可能なら noindex を付与する
- ユーザーが戻れる導線を置く

7. XSS リスクのある描画を縮小する
- `dangerouslySetInnerHTML` を使わず plain text ベースに寄せる
- HTML を扱うならサニタイズポリシーを明示する

8. ビルドとテストを誰でも再現できる状態にする
- `npm ci` / `npm run build` が通る
- Python venv と `pytest` が揃う
- README または手順書に実行コマンドを固定化する

## ビルドしてテスト状態で書くにすべき項目

このセクションは、手動で公開確認するためのチェック項目と確認方法をまとめる。

### A. フロントエンドのビルド成功確認

確認項目:
- 依存関係が正しく入る
- 本番ビルドが成功する
- 生成物に必要アセットが含まれる

確認方法:
1. `apps/frontend` で依存関係をクリーンインストールする
2. `npm run build` を実行する
3. エラーなく完了することを確認する
4. `dist` 配下に `index.html`、JS、CSS、manifest、icon 類が出力されることを確認する

手動チェックポイント:
- `vite` / `esbuild` 欠落がない
- PWA icon 参照切れがない
- `/vite.svg` のままになっていない

### B. バックエンドのテスト実行確認

確認項目:
- Python 実行環境が固定されている
- `pytest` が実行できる
- 既存テストが成功する

確認方法:
1. `.venv` もしくは指定の Python 環境を有効化する
2. `apps/backend/requirements.txt` を導入する
3. `python -m pytest` を実行する
4. 失敗時は依存不足かテスト失敗かを切り分ける

手動チェックポイント:
- `pytest` 未導入で止まらない
- `pyenv` 依存の曖昧な実行にならない
- CI と同じ Python バージョンが使われる

### C. 認可テスト

確認項目:
- student が teacher 用 API に入れない
- teacher でも担当外 student の日誌は見えない
- teacher が担当 student のみ閲覧できる

確認方法:
1. student アカウントで `/diary/teacher/dashboard` を叩き、403 になることを確認
2. teacher アカウントで担当外 `student_id` の履歴 API を叩き、403 または 404 になることを確認
3. teacher アカウントで担当 student のみ取得できることを確認
4. 一覧 API に他クラスの氏名・メールアドレスが混ざらないことを確認

手動チェックポイント:
- ルーターで拒否できる
- サービス層でも抜け道がない
- 直接 API 叩きでも突破できない

### D. エラーハンドリング確認

確認項目:
- 500 系エラーで内部詳細を返さない
- 画面上には一般化された文言のみ出る
- サーバーログ側には原因が残る

確認方法:
1. 日誌生成 API で意図的に失敗を起こす
2. ブラウザ表示メッセージを確認する
3. レスポンス JSON の `detail` を確認する
4. サーバーログにのみ詳細が出ることを確認する

手動チェックポイント:
- 例外文字列がそのままブラウザに出ない
- SQL 名、トークン検証失敗詳細、外部 API 失敗詳細が漏れない

### E. 認証情報保護確認

確認項目:
- access token / refresh token の保存場所が意図通り
- 不要な二重保存がない
- サインアウト時に確実に消える

確認方法:
1. ログイン前後で `localStorage` / Cookie / session storage を確認する
2. `auth_tokens` の有無を確認する
3. サインアウト後に認証情報が残らないことを確認する
4. 複数タブでログアウト・再ログイン挙動を確認する

手動チェックポイント:
- トークンが複数キーに重複保存されていない
- 期限切れやリフレッシュ失敗時に不整合がない

### F. CORS / オリジン設定確認

確認項目:
- 公開 URL から API に正常アクセスできる
- 許可外オリジンからは拒否される
- 開発・デモ・本番の設定が混ざっていない

確認方法:
1. 本番予定ドメインからログイン、API 呼び出し、日誌送信を実行する
2. 開発用ドメイン、ローカル、別ドメインからの挙動差を確認する
3. preflight が通るかをブラウザ devtools で見る
4. レスポンスヘッダの `Access-Control-Allow-Origin` を確認する

手動チェックポイント:
- `demo.tanqmates.org` 固定のまま本番投入しない
- バックエンド CORS と nginx 設定が一致している

### G. メタデータ / OGP / アイコン確認

確認項目:
- title / description が適切
- canonical がある
- OGP / Twitter Card がある
- favicon / apple-touch-icon / PWA icon が参照切れしない

確認方法:
1. 本番ビルド後の HTML を確認する
2. ブラウザで favicon 表示を確認する
3. OGP テスターや SNS プレビューで見え方を確認する
4. `public` 配下のアイコン実体と参照パスを突き合わせる

手動チェックポイント:
- `/vite.svg` を本番で使わない
- OGP 画像 URL が絶対 URL になっている

### H. 404 / エラーページ確認

確認項目:
- 存在しない URL で適切な Not Found 表示になる
- ユーザーがトップへ戻れる
- SEO 的に不適切なインデックスを防げる

確認方法:
1. 存在しないパスへ直接アクセスする
2. SPA ルーティングとサーバー応答の両方を確認する
3. 404 ページに戻り導線があるか確認する

手動チェックポイント:
- 何でも `/` に戻すだけの状態をやめる
- 404 と正常ページの区別がつく

### I. XSS 観点の確認

確認項目:
- ユーザー入力や AI 出力が HTML として注入されない
- `dangerouslySetInnerHTML` 利用箇所に妥当な防御がある

確認方法:
1. 日誌編集欄に HTML 風文字列を入れて表示を確認する
2. 保存・再表示時に script や event handler が解釈されないことを確認する
3. 他の HTML 注入系箇所も横断検索して確認する

手動チェックポイント:
- plain text として扱われる
- もし HTML を許すなら必ずサニタイズが入る

## 公開前の推奨優先順位

### 公開を止めるべき項目

1. teacher API の認可不足を修正
2. 500 エラー詳細露出を修正
3. 本番 CORS / オリジン設定を修正

### 公開前に完了推奨の項目

4. トークン保存方針を見直し
5. OGP / icon / canonical を追加
6. 404 ページを追加
7. ビルドとテストを再現可能にする

### 追って改善でもよいが公開前に認識すべき項目

8. `dangerouslySetInnerHTML` 周辺の設計見直し
9. 公開後の監視、通知、障害検知の整備

## 今回の確認で実行できなかったこと

以下は環境都合で未完了:

- フロントエンド本番ビルドの成功確認
  - `vite` 実行不可
  - `esbuild` 欠落
- バックエンド pytest 実行
  - `.venv` に `pytest` 未導入

そのため、本レポートは主に静的レビュー結果であり、最終的な公開可否判断には手動ビルド・手動テストの実施が前提となる。

## 結論

現時点の探Qメイトは、公開前チェックの観点ではまだ「公開可」とは言いにくい。

特に、先生向け日誌 API の認可不足は即修正対象であり、これを解消するまでは本番公開を止めるべきである。その上で、エラー露出、CORS、本番メタデータ、ビルド再現性を整え、最後に手動の公開前確認を一通り通すのが妥当である。
