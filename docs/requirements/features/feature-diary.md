## 1. 概要

本ドキュメントは、探Qメイトに追加する「探究日誌機能」のMVP要件を定義する。

この機能の目的は次の3点。

- 生徒が自分の探究の変化・成長を言語化する機会を作ること
- 先生が生徒の思考プロセスを「読み物」として楽しみながら把握できること
- AIの解釈と生徒の自己認識のズレを記録し、将来の分析に活用できること

> **Note:** LangGraphは現時点で未実装。本要件はLangGraphなしで動作するMVPを対象とする。

---

## 2. ユーザーと利用文脈

### 2-1. ユーザー

| ユーザー | 役割 | 主な利用場面 |
|---|---|---|
| 生徒 | 日誌の生成・編集・提出 | 探究セッション終了後 |
| 先生 | 日誌の閲覧・コメント記入 | 放課後・授業準備時 |

### 2-2. 利用の前提

- 生徒は探Qメイトのチャット画面で探究に関する会話を行う
- 会話の内容が日誌生成の「元ネタ」となる
- 日誌生成のトリガーは生徒が手動で行う（1ボタン）
- 生成された日誌は生徒が確認・編集してから先生に送信される

## 3. 全体ワークフロー

### 3-0. 概要

| ステップ | 主体 | 内容 |
|---|---|---|
| ① ボタンを押す | 生徒 | チャット画面下部の「日誌を生成する」ボタンを押す |
| ② ログ取得・整形 | システム | 当日の対話ログと前回日誌をDBから取得する |
| ③ API リクエスト | システム | Anthropic API に送信し、JSON の下書きを生成する |
| ④ 下書き表示 | システム | 生成結果をUIに表示し、ai_draft をDBに保存する |
| ⑤ 編集 | 生徒 | 本文・引用・タグを加筆・修正・削除する |
| ⑥ 感情評価 | 生徒 | 手ごたえ・気分タグ・自由記述を入力する |
| ⑦ 提出 | 生徒 | 「提出する」ボタンを押してDBに書き込む |
| ⑧ 先生閲覧 | 先生 | ダッシュボードで日誌・感情評価を閲覧する |

> **Note:** ステップ③の生成はAnthropic APIへの単一リクエストで実現する（LangGraphは使わない）。

### 3-1. ステップ①：「日誌を生成する」ボタンを押す

**主体：生徒**

| レイヤー | 内容 |
|---|---|
| UI | チャット画面下部の固定ボタンを押す。会話が0ターンの場合はボタンが非活性（disabled）。 |
| システム | 会話ログが1件以上存在するかをチェック。存在する場合、日誌生成フローを起動する。 |
| データ | トリガーイベントのみ。この時点ではDBへの書き込みは発生しない。 |

| | 内容 |
|---|---|
| Input | チャット会話ログ（当日分） |
| Output | 日誌生成フロー起動 |
| エラー／例外 | ログ0件 → ボタン非活性、フロー起動しない |

### 3-2. ステップ②：当日の対話ログを取得・整形する

**主体：システム**

| レイヤー | 内容 |
|---|---|
| UI | ローディングインジケーターを表示する。「日誌を生成しています…」のテキストを出す。 |
| システム | DBから当日分の会話ログを全件取得。前回の日誌（`published_body`）も取得し、プロンプトに渡す準備をする。 |
| データ | `conversations` テーブル（当日分）と `diary_entries` テーブル（最新1件）を読み取り。 |

| | 内容 |
|---|---|
| Input | `student_id`、当日の日付 |
| Output | 整形済みログテキスト、前回日誌テキスト |
| エラー／例外 | DB接続エラー → エラーメッセージ表示、リトライボタン |

### 3-3. ステップ③：Anthropic API へリクエストを送信する

**主体：システム**

| レイヤー | 内容 |
|---|---|
| UI | ローディング継続。タイムアウトは15秒。 |
| システム | システムプロンプト＋ログテキストをAPIに送信。モデル: `claude-sonnet-4-6`、出力形式: JSONのみ。レスポンスを受け取り、JSONをパースする。 |
| データ | APIリクエスト: `system_prompt` / `conversation_logs` / `previous_diary` → APIレスポンス（JSON）: `draft_body` / `quote` / `quote_context` / `closing_question` / `suggested_tags` / `turning_point_detected` / `turning_point_note` |

| | 内容 |
|---|---|
| Input | 整形済みログ、前回日誌、システムプロンプト |
| Output | draft JSON（7フィールド） |
| エラー／例外 | APIエラー・タイムアウト → エラー表示、リトライボタン。JSONパース失敗 → 同上。 |

### 3-4. ステップ④：下書きをUIに表示する

**主体：システム**

| レイヤー | 内容 |
|---|---|
| UI | ローディング終了。下書きカードを表示する。本文（テキストエリア、編集可）・引用ブロック（別フィールド、編集可）・問いかけ（読み取り専用）・タグ（追加・削除可）・転換点フラグ（`turning_point_detected: true` の場合のみ） |
| システム | 生成した draft JSON をフロントエンドの状態として保持する。保存は提出時に行う。 |
| データ | この時点ではDBへの書き込みは発生しない。 |

| | 内容 |
|---|---|
| Input | draft JSON |
| Output | UIに下書きカード表示 |
| エラー／例外 | 下書き生成失敗 → エラーメッセージを表示し、AI見立てなしで記録を書ける導線を出す。 |

> **Note:** 下書きだけでは先生に共有されず、プロファイル形成用の観測も提出済み日誌とは区別して扱う。

### 3-5. ステップ⑤：生徒が下書きを確認・編集する

**主体：生徒**

| レイヤー | 内容 |
|---|---|
| UI | 本文テキストエリアで自由に加筆・修正・削除。引用フィールドも変更可能。タグを追加・削除できる。「送信しない」ボタンも常に表示（強制しない）。 |
| システム | 編集はフロントエンドの状態管理のみ。この時点ではAPIもDBも呼ばない。 |
| データ | 編集内容はメモリ（フロント state）に保持。DBへの書き込みは発生しない。 |

| | 内容 |
|---|---|
| Input | `ai_draft`（初期値として表示） |
| Output | 編集済み本文 state、編集済み引用 state、編集済みタグ state |
| エラー／例外 | 「送信しない」を選択 → フローを中断。`ai_draft` はDBに残る。 |

### 3-6. ステップ⑥：感情評価を入力する

**主体：生徒**

| レイヤー | 内容 |
|---|---|
| UI | 下書きカードの下部に感情評価UIを表示。手ごたえ（5段階ボタンまたはスライダー）・気分タグ（複数選択チップ）・自由記述（任意テキスト）。どれも任意。入力せずに提出もできる。 |
| システム | 感情評価もフロントエンドの状態管理のみ。「提出する」ボタンが活性化する。 |
| データ | `emotion` state として保持: `{ effort_score: 1–5, mood_tags: [], free_text: "" }` |

| | 内容 |
|---|---|
| Input | 生徒の入力 |
| Output | `emotion` state |
| エラー／例外 | 未入力のまま提出 → `emotion` は `null` として保存（許容） |

---

### 3-7. ステップ⑦：「提出する」ボタンを押す

**主体：生徒**

| レイヤー | 内容 |
|---|---|
| UI | 「提出する」ボタンを押す。送信中はローディング。成功後は「提出しました」の完了メッセージを表示する。同じ日でも必要なときに再度作成できる。 |
| システム | フロント state（編集済み本文・引用・タグ・emotion）をまとめてDBに新規保存する。差分量（`diff_added` / `diff_removed`）を計算して保存する。 |
| データ | `diary_entries` に INSERT: `published_body` / `published_quote` / `published_tags` / `emotion` / `diff_added` / `diff_removed` / `submitted_at` / `turning_point` を書き込む。 |

| | 内容 |
|---|---|
| Input | 編集済み本文、編集済み引用、タグ、`emotion`、`ai_draft`（差分計算用） |
| Output | `diary_entries` 更新完了 |
| エラー／例外 | DB書き込みエラー → エラーメッセージ、再送ボタン |

**差分量の計算方法**

```
diff_added   = max(0, len(published_body) - len(ai_draft.draft_body + ai_draft.quote))
diff_removed = max(0, len(ai_draft.draft_body + ai_draft.quote) - len(published_body))
```

> **Note:** 差分の「中身（何を変えたか）」は保存しない。文字数のみ記録する（心理的安全性の確保）。

**主体：先生**

| レイヤー | 内容 |
|---|---|
| UI | **クラスビュー**：提出済み生徒の一覧に追加。`turning_point: true` の場合は転換点バッジを付与。**個人ビュー**：日誌本文・引用ブロック・タグ・転換点フラグ・感情評価（手ごたえ・気分タグ）を表示。 |
| システム | 先生がダッシュボードを開いたタイミングで `diary_entries` を読み取り表示する（ページ読み込み時）。 |
| データ | `diary_entries` から読み取るフィールド: `published_body` / `published_quote` / `published_tags` / `emotion` / `turning_point` / `submitted_at`。**`ai_draft` / `diff_*` は先生には非公開。** |

| | 内容 |
|---|---|
| Input | `diary_entries`（当日分・提出済み） |
| Output | クラスビュー更新、個人ビューに日誌表示 |
| エラー／例外 | 未提出の生徒 → クラスビューに「未提出」バッジを表示 |

## 4. 機能要件

### 4-1. 生成トリガー（生徒側）

- チャット画面の下部に「今日の探究を日誌にする ✦」ボタンを常時表示する
- ボタンは会話が1ターン以上存在する場合のみ活性化する
- 1日に複数回押すことができる
- すでに当日の日誌を提出していても、生徒が必要だと思ったタイミングで追加の日誌を作成できる

### 4-2. 日誌の下書き生成

- その日の会話ログ全件をAnthropic APIに渡してJSONで下書きを生成する
- 生成されるJSONのフィールドは以下の通り

| フィールド | 型 | 説明 |
|---|---|---|
| `draft_body` | string | 下書き本文（100〜150字、引用を除く） |
| `quote` | string | ユーザー発話の原文引用（1箇所） |
| `quote_context` | string | 引用の文脈（10字以内） |
| `closing_question` | string | 生徒への問いかけ（1文） |
| `suggested_tags` | string[] | テーマタグ（2〜3個） |
| `turning_point_detected` | boolean | 転換点の有無 |
| `turning_point_note` | string | 転換点の説明（trueの場合のみ） |

- 前回の日誌テキストを参照として渡し、同じエピソードの繰り返しを防ぐ
- 生成失敗時はエラーメッセージを表示し、リトライボタンを出す

### 4-3. 下書きの表示・編集（生徒側）

- 下書き本文・引用・問いかけ・タグを画面に表示する
- 本文は自由に加筆・修正・削除できるテキストエリアとして表示する
- 引用は別フィールドで表示し、変更することができる
- タグは追加・削除できる
- 「送信しない」選択肢を常に用意する（強制送信しない）

### 4-4. 感情・手ごたえの評価（生徒側）

- **手ごたえ：** 5段階スライダーまたはボタン（1〜5）
- **気分タグ：** 複数選択可のチェックボックス
  - 楽しかった / もやもやした / 悔しかった / 驚いた / 疲れた / すっきりした / 不安だった / ワクワクした
- **自由記述欄：** 任意、1〜2文程度
- 感情評価は先生には直接見えない（集計・傾向として将来的に活用）

> **Note:** 感情データの先生への表示方法はMVP後のフェーズで設計する。

### 4-5. 先生ダッシュボード

- **クラスビュー：** 今日日誌を提出した生徒の一覧と要フォローフラグを表示する
- **個人ビュー：** 1名の日誌を以下の形式で表示する
  - 日誌本文（生徒が編集した最終版）
  - 引用ブロック（生徒の言葉をそのまま表示）
  - テーマタグ
  - 転換点フラグ（`turning_point_detected: true` の場合）
  - **感情評価：** 手ごたえ（5段階）と気分タグを表示する
- ~~先生がコメントを書き込めるテキストエリアを日誌の下に配置する~~ → **MVP対象外**

## 6. データ設計

### 6-1. テーブル：`diary_entries`

| フィールド | 型 | 説明 |
|---|---|---|
| `id` | UUID | プライマリキー |
| `student_id` | UUID | 生徒ID（外部キー） |
| `date` | DATE | 日誌の日付 |
| `ai_draft` | JSONB | AI生成の下書き（生徒のみ閲覧） |
| `published_body` | TEXT | 生徒が編集した最終版本文 |
| `published_quote` | TEXT | 生徒が選んだ引用 |
| `published_tags` | TEXT[] | 最終版タグ |
| `emotion` | JSONB | 感情評価（effort_score, mood_tags, free_text） |
| `diff_added` | INTEGER | 加筆文字数（差分量） |
| `diff_removed` | INTEGER | 削除文字数（差分量） |
| `turning_point` | BOOLEAN | AI判定の転換点フラグ |
| `submitted_at` | TIMESTAMPTZ | 送信日時 |

> **Note:** `diff_added` / `diff_removed` は文字数のみ記録する。何を変えたかの中身は保存しない（心理的安全性の確保）。

### 6-2. 差分データの設計方針

差分データは「探Qメイトのみが知っている情報」として扱う。

- 先生には表示しない
- 生徒にも表示しない
- 将来の活用例（MVP後フェーズ）
  - AIの解釈精度の改善指標
  - 生徒のメタ認知発達の指標
  - 編集量と探究フェーズの相関分析

## 7. AIプロンプト設計

### 7-1. 基本方針

- **生成モデル：** `claude-sonnet-4-6`
- **出力形式：** JSONのみ（マークダウンのコードブロックを含まない）
- 読み手は生徒本人（先生向けではなく、生徒が編集する下書きとして生成する）
- 解釈を一歩踏み込んで書く：生徒が「そうじゃなくて、本当はこういうこと」と言いたくなる余白を意図的に作る

### 7-2. 文体のルール

- 二人称（「あなたは」）、温かいが馴れ馴れしくない
- 「気づき」「成長」「素晴らしい」などの教育的ワードを避ける
- 評価的な褒め言葉を使わない
- 断定しない。「〜のように見えました」「〜かもしれません」を使う

### 7-3. 引用の選び方

- ユーザーの発言から1つ選ぶ（AIの発言は引用しない）
- 迷いや驚き・思考の転換が現れている発話を優先する
- その人らしい言い回しが出ている発話を優先する

### 7-4. プロンプト本体

```
あなたは「探Qメイト」です。
ユーザーとの会話ログが渡されます。
その内容から、ユーザー自身が「編集・加筆」するための日誌の下書きを書いてください。

## あなたの役割
完成品ではなく、ユーザーが自分の言葉に直すための出発点を作ること。
「私はあなたの今日をこう読みました」というトーンで書く。

## 重要な方針
- 読み手はユーザー本人
- 解釈を一歩踏み込んで書く → 「そうじゃなくて、本当はこういうこと」と言いたくなる余白を意図的に作る
- 断定しない。「〜のように見えました」「〜かもしれません」

## 下書きの構成
1. 書き出し（1文）: 今日の「空気」を二人称で
2. 本文（3〜4文）: 何を考え、どこに引っかかり、何に気づいたか
3. 引用（1箇所）: ユーザー自身の言葉をログからそのまま抜粋
4. 問いかけ（1文）: 「これは合っていますか？」という確認

## 文体のルール
- 二人称（「あなたは」）、温かいが馴れ馴れしくない
- 「気づき」「成長」「素晴らしい」などの教育的ワードを避ける
- 評価的な褒め言葉を使わない

## 引用の選び方
ユーザーの発言から1つ。迷いや驚き・思考の転換が現れているもの。
AIの発言は引用しないこと。

## 出力形式
JSON形式のみ。マークダウンのコードブロックも含めず、純粋なJSONのみ返すこと。

{
  "draft_body": "（下書き本文。引用を除く、100〜150字）",
  "quote": "（ユーザー発話の原文引用）",
  "quote_context": "（引用の文脈を10字以内で）",
  "closing_question": "（ユーザーへの問いかけ1文）",
  "suggested_tags": ["タグ1", "タグ2"],
  "turning_point_detected": false,
  "turning_point_note": ""
}
```
