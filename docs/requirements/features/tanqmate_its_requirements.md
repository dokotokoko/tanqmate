# 探Qメイト ITSモデル化 要件定義

作成日: 2026-05-24  
文書種別: 要件定義 / 設計方針ドラフト  
対象: 探Qメイトにおける ITS アーキテクチャ、ハーネス設計、リフレクション、観測プロファイル

---

## 1. この文書の目的

本ドキュメントは、探究学習支援AI「探Qメイト」を、単なるチャットアプリではなく、探究学習に特化した ITS（Intelligent Tutoring System）として実装するための要件を整理するものである。

特に、以下を明確化する。

- 探Qメイトにおける ITS 構成要素
- 学習者モデル、教授モデル、ドメイン/課題モデル、評価/観測モデルの役割
- `/chat` を主導線に残しながら、内部では TutorOrchestrator 相当の制御を通す設計
- リフレクション機能のワークフロー
- AI視点リフレクションを蓄積する「観測プロファイル」の位置づけ
- 生徒の主体性を損なわない応答方針
- raw conversation log に依存しすぎない観測・記録方針

---

## 2. 探Qメイトの基本思想

探Qメイトは、生徒を効率的に正解へ誘導するためのAIではない。

探Qメイトは、生徒が自分の問いを見つけ、迷い、試し、振り返り、自分の言葉で探究を進めるための伴走者である。

そのため、探Qメイトの設計思想は以下とする。

### 2.1 生徒を動かすのではなく、動きたいときに動けるようにする

探Qメイトは、生徒に毎回次の行動を押しつける存在ではない。

探究学習では、迷う、保留する、違和感を持つ、寝かせる、戻る、といった時間にも意味がある。

したがって、探Qメイトは「次アクション」を毎回生成するのではなく、生徒が動きたくなったときに使える足場を置くことを重視する。

### 2.2 AIの性能を弱めるのではなく、AIの知性の出し方を制御する

探Qメイトは、LLMの性能をあえて落とすことを目的としない。

高性能なLLMの推論・整理・例示・批判的検討の力を、探究学習に合うように露出制御する。

重要なのは、答えを禁止することではなく、どのタイミングで、どこまで、どの形でAIの知性を見せるかである。

### 2.3 質問は目的ではなく、介入手段である

探Qメイトは「問いかけ中心」のAIではあるが、質問を増やすことを目的にしない。

質問は、生徒の思考を支えるための手段であり、質問だけが続くと生徒にとって負担になる。

原則として、質問の前には整理、例示、選択肢提示、受け止めなどの価値提供を行う。

### 2.4 変容はチャット内だけで判定しない

探究学習における変容は、チャットの直後に現れるとは限らない。

変容は、授業中の行動、友達や先生との会話、家で考えたこと、成果物の更新、リフレクション日誌などに現れる可能性がある。

そのため、探Qメイトはターン単位で生徒の変容を断定しない。

### 2.5 生徒をラベル化しない

探Qメイトは、生徒を「主体性が低い」「調査が苦手」「やる気がない」などと固定的にラベル化しない。

扱うのは、あくまでその時点の発話、感情選択、AI視点リフレクション、日誌、成果物から見える仮説的なサインである。

### 2.6 会話を監視するのではなく、探究プロセスを可視化する

先生・学校向けに raw conversation log をそのまま提示することは避ける。

代わりに、探究フェーズ、問い、仮説、迷い、支援プロセス、先生確認が必要な事項など、教育的に意味のある状態要約を提示する。

## 3. 用語定義

| 用語 | 定義 |
|---|---|
| ITS | Intelligent Tutoring System。学習者状態、教授方略、課題状態、観測情報をもとに支援を調整するシステム。 |
| ハーネスエンジニアリング | LLM単体の出力ではなく、状態、文脈、方略、ツール、検証、ログ、制御を含めてAIの振る舞いを設計すること。 |
| 探Qメイト | 探究学習におけるAI伴走者。問い・仮説・調査・振り返りを支援する。 |
| 学習者モデル | 現在の生徒に合った支援を行うための内部状態モデル。 |
| 教授モデル | どのような支援、発話、応答スタイル、開示レベルで介入するかを決めるモデル。 |
| ドメイン/課題モデル | 探究テーマ、問い、仮説、調査計画、成果物、日誌など、探究課題の現在地を表すモデル。 |
| 評価/観測モデル | 支援過程、感情選択、AI視点リフレクション、観測プロファイル、生徒日誌、応答リアクションなどを記録するモデル。 |
| AI視点リフレクション | リフレクション時に、AIが生徒に提示する「AIから見た今日の振り返り」。 |
| 観測プロファイル | AI視点リフレクションを生徒個人ごとに時系列で蓄積した、AI視点の観測データセット。 |
| 生徒日誌 | AI視点リフレクションを受けて、生徒が自由記述で作成する本人の振り返り。 |
| 観測差分 | AI視点リフレクションと生徒日誌の一致・ズレ・補足を扱う将来的な分析対象。 |
| 開示レベル | AIがどこまで答えや具体案を見せるかの段階。 |
| 行動促進の強さ | 生徒に次の行動をどの程度促すかの段階。 |

## 4. 全体アーキテクチャ

探Qメイトは、表面的には `/chat` を主導線として維持する。

ただし、内部では単純なチャット補完ではなく、TutorOrchestrator 相当の統合制御を通す。

```text
/chat
  ↓
入力受付
  ↓
状態整理
  ↓
支援方略選択
  ↓
文脈構築
  ↓
応答生成
  ↓
教育的検証
  ↓
応答提示
  ↓
支援過程記録
```

### 4.1 TutorOrchestrator の役割

TutorOrchestrator は、以下を統合的に制御する。

- 生徒発話の受け取り
- 現在の探究状態の整理
- 学習者モデルの参照
- 観測プロファイルの参照
- 支援タイプの選択
- 発話アクトの選択
- 応答スタイルの選択
- 開示レベルの調整
- 質問数の制御
- 行動促進の強さの調整
- 生成後の教育的チェック
- ログ・状態の保存

### 4.2 `/chat` の基本処理フロー

```text
1. 生徒の発話を受け取る

2. 現在の文脈を整理する
   - 何について話しているか
   - どの探究フェーズか
   - 生徒は進みたいのか、迷っているのか、受け止めてほしいのか
   - 直近の感情選択や観測プロファイルに関連する要素はあるか

3. 支援方略を選ぶ
   - 支援タイプ
   - 発話アクト
   - 応答スタイル
   - 開示レベル
   - 質問予算
   - 行動促進の強さ

4. 応答を生成する
   - 質問しすぎない
   - 必要なら例や選択肢を出す
   - 行動を押しつけない
   - 保留や迷いを認める

5. 教育的に検証する
   - 質問が多すぎないか
   - 答えを出しすぎていないか
   - 逆に材料不足になっていないか
   - 行動促進が強すぎないか
   - 生徒の状態に対して適切か

6. 支援過程を記録する
   - raw conversation log ではなく、支援プロセス中心に記録する
```

## 5. ITS構成要素

探QメイトのITS構成要素は、以下の4つを基本とする。

1. 学習者モデル
2. 教授モデル
3. ドメイン/課題モデル
4. 評価/観測モデル

ツール群は将来的に追加するが、現時点では保留扱いとする。

---

## 6. 学習者モデル

学習者モデルは、生徒に合った支援を行うための内部モデルである。

ただし、生徒の能力や性格を固定的に評価するものではない。

### 6.1 学習者モデルで保持する項目

| 項目 | 内容 |
|---|---|
| 学年帯 | 小学生、中学生、高校生、その他。語彙や説明粒度の調整に使う。 |
| 学校・先生からの制約 | 課題範囲、提出物、使用可能な調査方法、校外調査ルール、個人情報の扱いなど。 |
| 現在の興味・関心 | 生徒が発話や日誌で示したテーマ・関心。 |
| 自分ごと感のサイン | 自分の経験、学校生活、家庭、地域、好きなこと、困りごととの接続。 |
| 困り感・不満・迷いのサイン | 「わからない」「だるい」「質問ばっかり」「決められない」など。 |
| 支援の受け取り方 | 例があると考えやすい、選択肢があると選びやすい、自分で考えたい、など。 |
| 生徒の選択・保留・拒否 | 選んだ、理由を言った、修正した、保留した、拒否した、別方向を提案した、など。 |
| 直近の感情選択 | リフレクション時に生徒が選択した感情。 |
| 観測プロファイルへの参照 | AI視点リフレクションの蓄積から見える傾向を参照する。 |

### 6.2 学習者モデルから除外する項目

以下は、現時点では学習者モデルに含めない。

| 除外項目 | 理由 |
|---|---|
| 探究経験レベル | 学年と相関しやすく、学年以外に正確な計測が難しいため。 |
| 生徒の固定的能力評価 | ラベル化や誤推定のリスクが高いため。 |
| 性格診断的な項目 | 教育支援上の必要性が低く、監視感や決めつけにつながりやすいため。 |

### 6.3 学習者モデルの注意点

- 「この生徒はこういう人」と断定しない。
- 直近の発話・感情選択・日誌・成果物・観測プロファイルをもとに、支援の仮説を持つ。
- 観測プロファイルは参照するが、現在の生徒の発話を上書きしない。
- 保留や拒否も主体性の一形態として扱う。

---

## 7. 教授モデル

教授モデルは、探Qメイトがどのように支援するかを決めるモデルである。

### 7.1 教授モデルで保持する項目

| 項目 | 内容 |
|---|---|
| 支援タイプ | 今回、探究のどの部分を支援するか。 |
| 発話アクト | AIがどのような発話行為を行うか。 |
| 応答スタイル | AIがどのような役割・雰囲気で話すか。 |
| 開示レベル | AIがどこまで答えや具体案を出すか。 |
| 質問予算 | 1ターンで質問してよい量。原則1つまで。 |
| 行動促進の強さ | 生徒に次の行動をどの程度促すか。 |
| 待つ・受け止める方略 | 迷い、違和感、保留を尊重するための方略。 |
| 介入理由 | なぜその支援を選んだか。 |

---

## 8. 支援タイプ

支援タイプは、「今回、探究のどの部分を助けるか」を表す。

| 支援タイプ | 内容 |
|---|---|
| テーマ発見支援 | 何に興味があるかを見つける。 |
| 問いの改善支援 | 広すぎる問いや曖昧な問いを具体化する。 |
| 仮説づくり支援 | 調査前の予想や仮説を作る。 |
| 調査設計支援 | どう調べるかを考える。 |
| 根拠確認支援 | 主張と根拠がつながっているかを見る。 |
| 分析支援 | 調査結果をどう読み取るか支援する。 |
| 文章化支援 | レポート、発表、ポスターなどにまとめる。 |
| 振り返り支援 | 学び、迷い、気づき、変化を整理する。 |
| 感情・迷いの受け止め | 不安、不満、違和感、停滞感を受け止める。 |
| 先生への接続支援 | 先生の判断や確認が必要な状態に接続する。 |

### 8.1 「動機づけ支援」の扱い

「動機づけ支援」という名称は、AIが生徒を動かす・やる気にさせるニュアンスが強くなる可能性がある。

そのため、探Qメイトでは「感情・迷いの受け止め」という表現を優先する。

---

## 9. 発話アクト

発話アクトは、AIがそのターンで何をする発話なのかを表す。

| 発話アクト | 内容 |
|---|---|
| 言い換え | 生徒の発言を探究に使いやすい形に整える。 |
| 診断 | 問い、仮説、計画、成果物の状態を見立てる。 |
| 説明 | 探究の考え方、基準、型を説明する。 |
| 例示 | 具体例を出す。 |
| 選択肢提示 | 複数の方向性を出し、生徒が選びやすくする。 |
| 理由を問う | 生徒の選択や考えの理由を聞く。 |
| 前提を揺さぶる | 思い込みや弱い仮説に気づかせる。 |
| 要約 | ここまでの内容を短くまとめる。 |
| 進みたい場合の足場提示 | すぐに行動を促すのではなく、動きたいときの入口を示す。 |
| 振り返り | 学びや変化を自分の言葉にする支援。 |
| 励まし | 状態を認め、前進感を支える。 |
| 待つ | 今すぐ決めなくてよいことを認める。 |
| 受け止める | 違和感、迷い、不満、保留を否定せず扱う。 |

---

## 10. 応答スタイル

応答スタイルは、AIがどのような役割・雰囲気で話すかを表す。

| 応答スタイル | 内容 |
|---|---|
| 軽いソクラテス式 | 質問で考えを引き出すが、問い詰めない。 |
| コーチ型 | 状態を認めながら、必要に応じて前進の足場を示す。 |
| 研究パートナー型 | 上から評価せず、一緒に考える相棒として話す。 |
| ルーブリック評価者型 | 先生の評価基準に照らして改善点を見る。 |
| 直接説明型 | 必要な知識や方法をはっきり教える。 |
| 最小応答型 | 生徒が進めているとき、短く必要最低限に返す。 |
| 受け止め型 | 生徒の迷い、違和感、保留、不満を扱う。 |

---

## 11. 開示レベル

開示レベルは、AIがどこまで答えや具体案を見せるかを表す。

「答える / 答えない」の二択ではなく、段階的に制御する。

| レベル | 名称 | AIの出し方 |
|---:|---|---|
| 0 | 反映だけ | 生徒の考えを整理するだけ。 |
| 1 | 軽いヒント | 方向性だけ示す。 |
| 2 | 観点・選択肢 | 考えるための観点や選択肢を出す。 |
| 3 | 例・型 | 具体例やテンプレートを出す。 |
| 4 | 直接説明 | 必要な知識や手順をかなり明確に教える。 |
| 5 | 完成案に近い提示 | ほぼ答えや完成例を出す。使用は限定的。 |

### 11.1 開示レベルの注意点

- 高い開示レベルは、必ずしも悪ではない。
- 生徒が詰まっている場合、例や型を出すことは有効である。
- 完成案に近い提示は、比較用・添削用・先生許可ありの場合などに限定する。
- 開示レベルと行動促進の強さは別概念として扱う。

---

## 12. 質問予算

質問予算は、1ターンで質問してよい量を制御する。

### 12.1 基本方針

- 原則として、1ターンの実質的な質問は1つまでとする。
- 質問の前に、整理、例示、選択肢提示、受け止めなどの価値提供を行う。
- 生徒が「質問ばかり」と感じている場合は、質問を止め、例示や整理に切り替える。

### 12.2 悪い例

```text
なぜそう思いましたか？
どうやって調べますか？
誰に聞きますか？
いつやりますか？
```

### 12.3 良い例

```text
今のテーマは少し広いので、まず「誰のどんな行動を見るか」を決めると考えやすくなります。
たとえば、クラスメイト、先生、家族の3方向があります。
この中で、一番聞きやすそうなのはどれですか？
```

## 13. 行動促進の強さ

行動促進の強さは、生徒に次の行動をどの程度促すかを表す。

探Qメイトでは、毎回次アクションを出すことを義務化しない。

| レベル | 名称 | 内容 |
|---:|---|---|
| 0 | 行動を促さない | 迷い、違和感、保留を受け止める。 |
| 1 | 進みたくなったときの足場だけ置く | 今すぐ動かなくてもよいが、入口を示す。 |
| 2 | 小さな行動を提案する | 生徒が進みたそうなときに、具体的な一歩を提案する。 |
| 3 | 手順化する | 生徒が明確に進みたいとき、手順として整理する。 |

### 13.1 行動促進の注意点

- 行動を促さないことも、教育的支援である。
- 「次にこれをやろう」と毎回提示すると、主体性を損なう可能性がある。
- 生徒が「動きたい」と示したときに、初めて手順化を強める。

---

## 14. 探Qメイトの応答契約

探Qメイトの応答は、以下の契約に従う。

```text
1. 生徒の言葉や状態を丁寧に受け取る。
2. 必要なら、考える材料・例・選択肢を出す。
3. 質問は原則1つまでにする。
4. 行動を促すかどうかは、生徒の状態と意図に応じて決める。
5. 毎回の次アクションを義務化しない。
6. 生徒が動きたくなったときの足場は残してよい。
7. 保留・迷い・違和感も尊重する。
8. AIの見立ては断定せず、仮説として扱う。
```

### 14.1 応答例

生徒発話:

```text
環境問題について調べたいけど、なんか広すぎてよくわからない。
```

探Qメイト応答:

```text
「環境問題」は大事なテーマですが、そのままだと広くて、どこから考えればいいか迷いやすいです。

探究にしやすくするには、まず「自分の生活に近い場所」に寄せると考えやすくなります。

たとえば、

A. 学校の紙ごみを減らすには？
B. 給食の食品ロスはなぜ起きる？
C. 教室の電気使用量を減らすには？
D. 通学路のごみを減らす方法はある？

のようにできます。

今すぐ決めなくても大丈夫です。
この中で「ちょっと自分に近いかも」と思うものがあれば、そこから広げられます。
```

## 15. ドメイン/課題モデル

ドメイン/課題モデルは、探究そのものの現在地を表す。

探究学習では、テーマ領域が多様であり、数学のように正答が固定されているわけではない。

そのため、探Qメイトのドメインモデルは、個別知識よりも「探究プロセス」を中心に扱う。

### 15.1 探究フェーズ

| フェーズ | 内容 |
|---|---|
| テーマ探索 | 興味のあるテーマを探している。 |
| 問いづくり | 探究問いを作っている。 |
| 仮説づくり | 予想や仮説を作っている。 |
| 調査設計 | 調べ方を決めている。 |
| データ収集 | アンケート、観察、資料調査などをしている。 |
| 分析 | 集めた情報を読み取っている。 |
| 主張づくり | 根拠をもとに考えを組み立てている。 |
| 発表・表現 | レポート、スライド、ポスターなどにまとめている。 |
| 振り返り | 学びや改善点を整理している。 |

### 15.2 成果物

| 成果物 | 内容 |
|---|---|
| テーマ | 大まかな関心領域。 |
| 探究問い | 調べる中心の問い。 |
| 仮説 | 調査前の予想。 |
| 調査計画 | 誰に、何を、どうやって、どれくらい調べるか。 |
| 根拠マップ | 主張と根拠の対応。 |
| 成果物 | レポート、スライド、ポスター、プロトタイプなど。 |
| 生徒日誌 | 生徒自身の自由記述による振り返り。 |
| 保留中の論点 | まだ決めていないこと、違和感、迷い。 |

### 15.3 探究品質の観点

探究品質は、評価ラベルではなく、支援の方向性を考えるための観点として扱う。

| 観点 | 内容 |
|---|---|
| 具体性 | テーマや問いが具体的か。 |
| 調査可能性 | 実際に調べられるか。 |
| 実行可能性 | 時間、人数、環境的に実行できるか。 |
| 根拠との対応 | 主張と根拠がつながっているか。 |
| 独自性 | 自分なりの視点があるか。 |
| 自分ごと感 | 自分の経験や関心とつながっているか。 |
| 倫理リスク | 個人情報、差別、危険な調査などのリスクがあるか。 |

## 16. リフレクション機能のワークフロー

探Qメイトのリフレクションは、以下の3ステップで構成する。

```text
① 感情選択
   ↓
② AI視点リフレクションの提示
   ↓
③ 生徒が自由記述で日誌を作成
```

### 16.1 ① 感情選択

生徒は、その日の探究や学習に関する感情を選択する。

感情選択は、生徒の状態を決めつけるためではなく、AI視点リフレクションを生成するための入口として扱う。

例:

```text
- すっきりした
- 迷っている
- ちょっと進んだ
- だるい
- 不安
- 面白くなってきた
- よくわからない
- まだ決めたくない
```

### 16.2 ② AI視点リフレクションの提示

AIは、感情選択とその時点までの探究状態をもとに、「AIから見た今日の振り返り」を提示する。

これは生徒の真実を断定するものではなく、AI伴走者の仮説的な見立てである。

AI視点リフレクションは、生徒の日誌作成を支えるために提示される。

### 16.3 ③ 生徒が自由記述で日誌を作成

生徒は、AI視点リフレクションを読んだ後、自分の言葉で日誌を書く。

この日誌は、AIの見立てに従う必要はない。

生徒は、AIの見立てに同意してもよいし、違和感を書いてもよい。

---

## 17. AI視点リフレクション

AI視点リフレクションは、リフレクション工程の②で提示される。

### 17.1 AI視点リフレクションの目的

- 生徒が自分の探究を外側から見られるようにする。
- チャットや成果物だけでは見えにくい迷い・変化・感情を言語化する。
- 生徒が自由記述の日誌を書き始める足場にする。
- 探Qメイトがその時点でどう生徒を見立てたかを記録する。

### 17.2 AI視点リフレクションの構成

AI視点リフレクションは、以下の要素を含むことが望ましい。

```text
1. 今日の探究がどう見えたか
2. 今日少し動いたように見えること
3. まだ迷っていそうなこと
4. 選択された感情と探究状態のつながり
5. 生徒が日誌を書くための視点
6. 断定ではなく、仮の見立てであること
```

### 17.3 AI視点リフレクションの例

```text
今日のあなたの探究は、「環境問題」という大きなテーマから、少しずつ自分の生活に近い場所を探しているように見えます。

特に、学校や給食のように、自分が実際に見たことのある場面に近づくと、考えやすくなっているかもしれません。

一方で、まだ「何を調べるか」ははっきり決まっていないようにも見えます。

今日選んだ「迷っている」という感情は、テーマが悪いというより、まだ自分に近い切り口を探している状態に近いかもしれません。

この見立てが合っているかどうかも含めて、自分の言葉で今日の日誌に書いてみてください。
```

## 18. 観測プロファイル

観測プロファイルは、AI視点リフレクションを生徒個人ごとに蓄積した、AI視点の観測データセットである。

### 18.1 定義

観測プロファイルとは、リフレクション工程でAIが生徒に提示した「AI視点リフレクション」を時系列に蓄積した、AI伴走者側の観測データセットである。

### 18.2 観測プロファイルの主材料

観測プロファイルの主材料は、③の生徒日誌ではなく、②のAI視点リフレクションである。

```text
感情選択
  ↓
AI視点リフレクション
  ↓
日次AI観測レコードとして保存
  ↓
観測プロファイルに蓄積
```

### 18.3 観測プロファイルと生徒日誌の関係

| 項目 | 観測プロファイル | 生徒日誌 |
|---|---|---|
| 主体 | AI | 生徒 |
| 内容 | AIがどう見立てたか | 生徒が自分の言葉でどう振り返ったか |
| 生成タイミング | 日誌作成の前 | AI視点リフレクション提示の後 |
| 役割 | AI伴走者の観測履歴 | 生徒本人の自己記述 |
| 注意点 | AIの見立てを事実化しない | AIの見立てに従わせない |

### 18.4 観測プロファイルと学習者モデルの関係

観測プロファイルは、学習者モデルそのものではない。

学習者モデルは、現在の支援を調整するための内部状態である。

観測プロファイルは、これまでAIが生徒をどう見立ててきたかの時系列データセットである。

ただし、学習者モデルは必要に応じて観測プロファイルを参照する。

```text
観測プロファイル
  = AI視点リフレクションの蓄積

学習者モデル
  = 現在の支援調整に使う内部状態

関係
  = 学習者モデルが観測プロファイルを参照する
```

### 18.5 観測プロファイルの注意点

- 観測プロファイルは「生徒の真実」ではない。
- 「AIがその時点でどう見たか」の履歴である。
- AIの誤った見立ても蓄積される可能性がある。
- そのため、生徒を断定する表現ではなく、AI観測として保存する。
- 必要に応じて、後から生徒日誌とのズレを扱えるようにする。

## 19. 日次AI観測レコード

日次AI観測レコードは、1回分のAI視点リフレクションを保存する単位である。

### 19.1 保存項目

```text
日次AI観測レコード

- 観測レコードID
- 生徒ID
- 日付
- 対象期間
- 感情選択
- その日の探究フェーズ
- 参照した状態スナップショットID
- AI視点リフレクション本文
- AIが見た今日の探究
- AIが見た変化
- AIが見た迷い・引っかかり
- AIが見た感情・熱量のサイン
- AIが見た支援の受け取り方
- AIが見たメタ認知の芽
- 見立ての確信度
- 生成方針バージョン
- 使用モデル
- 生成日時
- 生徒に提示されたか
- 紐づく生徒日誌ID
```

### 19.2 保存タイミング

AI視点リフレクションは、生徒に提示した時点で保存する。

生徒日誌の作成後に上書きしない。

生徒日誌を受けて解釈を追加する場合は、別レコードとして保存する。

## 20. 観測プロファイルの2層構造

観測プロファイルは、以下の2層で考える。

### 20.1 観測ログ型プロファイル

日次AI観測レコードを時系列で蓄積したもの。

これは、AIが各日でどう生徒を見立てたかの履歴である。

例:

```text
2026-05-01:
AIは、テーマがまだ広く、具体化に迷っているように見立てた。

2026-05-02:
AIは、学校生活に近いテーマに寄せると考えやすそうだと見立てた。

2026-05-03:
AIは、調査方法に進む場面で少し負担感があるように見立てた。
```

### 20.2 集約型プロファイル

観測ログ型プロファイルから、繰り返し見られる傾向を要約したもの。

例:

```text
この生徒について、AIが継続的に見立てていること

- 大きなテーマを、学校生活に近い具体例へ寄せると考えやすそう。
- 質問だけよりも、例や選択肢がある方が言葉が出やすそう。
- 調査方法を決める段階で負担感が出やすそう。
```

### 20.3 集約型プロファイルの注意点

集約型プロファイルは便利だが、生徒を固定的に扱うリスクがある。

そのため、初期実装では観測ログ型プロファイルを優先し、集約型プロファイルは後続フェーズで慎重に実装する。

## 21. 生徒日誌

生徒日誌は、AI視点リフレクションを受けて、生徒が自由記述で作成する記録である。

### 21.1 生徒日誌の役割

- 生徒自身の言葉による振り返りを残す。
- AIの見立てに同意する、違和感を書く、補足するなどを可能にする。
- 探究の行動・感情・気づきを本人の言葉で記録する。

### 21.2 生徒日誌の保存項目

```text
生徒日誌

- 日誌ID
- リフレクションセッションID
- 生徒ID
- 日付
- 自由記述本文
- 作成日時
```

### 21.3 生徒日誌の注意点

- AI視点リフレクションに従わせない。
- 生徒がAIの見立てに反論・修正・違和感を示せるようにする。
- 生徒日誌は観測プロファイルの主材料ではない。
- ただし、将来的には観測差分の材料になり得る。

## 22. 観測差分

観測差分は、AI視点リフレクションと生徒日誌の一致・ズレ・補足を扱う概念である。

初期実装では必須ではないが、将来的に重要な分析対象となる。

### 22.1 観測差分の例

AI視点リフレクション:

```text
今日はテーマが自分の生活に近づいてきたように見えます。
```

生徒日誌:

```text
たしかに、給食の残食なら自分の学校で見たことがあるので考えやすかった。
```

観測差分:

```text
AIの見立てと生徒の自己記述が一致している。
```

別の例:

AI視点リフレクション:

```text
今日はテーマが自分の生活に近づいてきたように見えます。
```

生徒日誌:

```text
自分に近いというより、先生に言われたから給食にしただけ。
```

観測差分:

```text
AIは自分ごと化と見立てたが、生徒は外部要因による選択と記述している。
```

### 22.2 観測差分の注意点

- 観測差分は、生徒を評価するためではなく、AIの見立てを検証するために使う。
- AIの観測精度向上や支援方略調整に利用できる。
- 初期段階では保留してよい。

## 23. 評価/観測モデル

評価/観測モデルは、探Qメイトがどのような支援を行い、生徒がどのように反応し、AIがどう見立てたかを記録するモデルである。

ただし、「評価」という言葉には注意が必要である。

探究学習では、短期的なターン単位で生徒の変容を断定することは難しい。

そのため、評価/観測モデルは、学習者を評価する装置ではなく、支援過程と観測仮説を蓄積する装置として扱う。

### 23.1 評価/観測モデルで保持する項目

| 項目 | 内容 |
|---|---|
| 支援過程記録 | どんな支援タイプ、発話アクト、応答スタイル、開示レベルだったか。 |
| 感情選択 | リフレクション開始時に生徒が選んだ感情。 |
| AI視点リフレクション | AIが提示した日次の見立て。 |
| 観測プロファイル | AI視点リフレクションの蓄積。 |
| 生徒日誌 | 生徒が自由記述で作成した振り返り。 |
| 応答リアクション | チャット応答に対する生徒の反応。 |
| 成果物変化 | テーマ、問い、仮説、調査計画、成果物の変化。 |
| 運用品質ログ | 応答時間、エラー、コストなど。教育評価とは分ける。 |

### 23.2 応答リアクション

応答リアクションは、探Qメイトの支援が生徒にどう受け取られたかを示す重要なシグナルである。

例:

| 生徒の反応 | 解釈 |
|---|---|
| だるい | 支援が重かった、質問が多すぎた、開示レベルが低すぎた可能性。 |
| 質問ばっかり | 質問予算の調整が必要。 |
| もっと例がほしい | 開示レベルを上げる、例示を増やす。 |
| すごくわかりやすい | 支援が合っていた可能性。 |
| 何をすればよいか迷いが晴れた | 整理・選択肢提示・足場提示が有効だった可能性。 |
| いったん考えたい | 低評価ではなく、保留や主体性のサインとして扱う。 |

### 23.3 運用品質ログの扱い

応答時間、文字量、トークン数、APIコスト、エラー率などは、プロダクト運用上は重要である。

ただし、これらを教育的評価として扱わない。

```text
応答が短い = 良い
応答が長い = 悪い
応答が速い = 良い
応答が遅い = 悪い
```

とは限らない。

運用品質ログは、教育的観測とは分離して保存する。

チャットターンごとに、支援プロセス中心のログを保存する。

raw conversation log を先生・学校向けにそのまま出すことは避ける。

```text
チャットターンログ

- ターンID
- セッションID
- 生徒ID
- 応答前状態スナップショットID
- 応答後状態スナップショットID
- 支援タイプ
- 発話アクト
- 応答スタイル
- 開示レベル
- 質問予算
- 行動促進の強さ
- 介入理由コード
- 介入理由要約
- 応答リアクション分類
- 教育的検証結果
- 応答時間
- エラーコード
- 生成方針バージョン
- 使用モデル
```

### 24.2 介入理由コード

介入理由は自由記述だけでなく、分析しやすいコードとして保存する。

| 介入理由コード | 内容 |
|---|---|
| 問いが曖昧 | 問いが広すぎる、ぼんやりしている。 |
| 前進感が低い | 会話や探究が進んでいない。 |
| 不満・困り感が高い | 生徒が困惑、負担、不満を示している。 |
| 例示が必要 | 抽象説明だけでは進みにくい。 |
| 丸投げリスク | AIに任せすぎている可能性。 |
| 順調に進んでいる | 余計な介入を減らすべき。 |
| 評価基準とのズレ | 先生のルーブリックとズレている。 |
| 安全・プライバシー懸念 | 個人情報、倫理、危険な調査など。 |
| 先生確認が必要 | AIだけで判断しない方がよい。 |
| 保留・違和感の尊重 | 今すぐ進めるより、迷いを扱うべき。 |

## 25. 教育的検証

応答生成後、必要に応じて教育的検証を行う。

### 25.1 検証項目

| 検証項目 | 内容 |
|---|---|
| 質問が多すぎないか | 実質的な質問が複数連続していないか。 |
| 答えを出しすぎていないか | 生徒の思考を置き換える形になっていないか。 |
| 材料が少なすぎないか | 生徒に考える材料を渡さず質問だけしていないか。 |
| 行動促進が強すぎないか | 毎回行動を押しつけていないか。 |
| 保留を認めているか | 迷い、違和感、保留を尊重できているか。 |
| 学年に合っているか | 語彙や説明の粒度が適切か。 |
| 先生確認が必要か | 倫理・安全・学校ルール上、先生確認が必要か。 |

```text
- 調査計画の有無

- 生徒が日誌で書いた気づきの要約

- 支援が合いにくそうだった場面
- 先生確認が必要そうなこと
```

### 27.2 先生向けに避ける情報

```text
- raw conversation log の常時閲覧
```

## 28. プライバシー・安全設計

探Qメイトは、学校現場で使われることを前提に、プライバシーと心理的安全性を重視する。

### 28.1 基本方針

- raw conversation log を先生・学校向けにそのまま出さない。
- 研究・実証用途でも、個人を直接読めるログではなく、状態推定と支援プロセス中心の記録を優先する。
- AIの見立ては「事実」ではなく「見立て」として扱う。
- 生徒の感情や主体性を断定的に評価しない。
- 先生確認が必要な倫理・安全上のリスクを検出する。

### 28.2 倫理・安全上の注意領域

```text
- 個人情報を扱う調査
- 家庭環境、病気、体重、収入などセンシティブな情報
- 誰かを責める形の調査
- 校外調査
- 無断撮影・録音
- 差別や偏見につながる問い
- 危険な実験や行動
```

### 29.3 初回MVPの実装決定

初回MVPでは、`/chat` の内部ITS化、支援プロセスログ保存、観測型プロファイル、集約型プロファイルを実装対象とする。

リフレクション3ステップは既存の日誌機能を指すため、新しい日誌UIや公開APIは追加しない。既存の日誌生成の裏側で、AI日誌下書きをAIによる見立てとして観測型プロファイルに蓄積する。

生徒UIには従来どおりチャット本文とクエストカードのみを表示し、支援タイプ、発話アクト、開示レベル、行動促進の強さなどのITSメタデータは内部ログとして保存する。

raw chat log は従来どおり保存する。ただし、チャット履歴以外のUI、特に先生向けUIには表示しない。

先生向けに表示する情報は、生徒が提出した日誌と共有用summaryを基本とする。AI見立て、観測プロファイル、支援プロセスログはMVPでは表示しない。

TutorOrchestrator の方略選択は、ルールベースとLLM判定のハイブリッドとする。明らかに悪い体験を防ぐ判定は高速なルールで行い、曖昧な意図、複合状態、感情・迷い・探究フェーズの読み取りが難しいケースのみLLM判定を使う。

実装上も、学習者モデル、教授モデル、ドメイン/課題モデル、評価/観測モデルを内部型として明示的に構築する。TutorOrchestrator は raw な発話だけでなく、この4モデルのスナップショットを参照して方略選択する。チャットターンログには raw conversation log ではなく、4モデルの内部スナップショット、支援方略、教育的検証結果を保存する。

観測プロファイルは2層で扱う。観測型プロファイルは既存日誌機能のAIによる見立てを時系列に蓄積した正本であり、集約型プロファイルは観測型プロファイルをもとに作成した分析・要約の正本である。

「先生確認が必要な事項」は初回MVPには含めない。

## 30. 非機能要件

### 30.1 追跡可能性

各応答について、以下を追跡できること。

```text
- どの状態を見て
- どの支援タイプを選び
- どの発話アクトを使い
- どの開示レベルで
- どの行動促進の強さで
- なぜその応答をしたのか
```

### 30.2 方針バージョン管理

AI視点リフレクション、チャット応答、教育的検証には、生成方針バージョンを持たせる。

これにより、後から「どの方針で生成された見立てか」を追跡できるようにする。

### 30.3 モデル非依存性

LLMモデルは将来的に変更される可能性がある。

そのため、探Qメイトの中核価値は、特定モデルのプロンプトではなく、ハーネス、状態管理、観測プロファイル、教授方略の制御に置く。

### 30.4 説明可能性

探Qメイトの支援方略は、内部的に説明可能であること。

少なくとも、開発者・研究者・学校側に対して、支援タイプ、発話アクト、開示レベル、行動促進の強さ、介入理由を説明できること。

## 32. 探Qメイトの最終的な位置づけ

探Qメイトは、LLMが回答するチャットアプリではない。

探Qメイトは、生徒の探究状態を整理し、問い・仮説・調査・成果物・振り返りを支援し、AI視点リフレクションと観測プロファイルを通じて、生徒が自分の探究をメタ認知するための伴走型ITSである。

その中核は、以下にある。

```text
- 生徒を動かすのではなく、動ける状態を作る
- 質問ではなく、支援方略を設計する
- 答えを禁止するのではなく、AIの知性の開示を制御する
- チャット内の即時反応だけでなく、リフレクションを重視する
- 生徒日誌とAI視点リフレクションを分けて保持する
- 観測プロファイルを、AI伴走者の見立ての蓄積として扱う
- 生徒をラベル化せず、見立てを仮説として扱う
```
