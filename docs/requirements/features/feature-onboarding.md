# 探Qメイト 初回体験再設計 要件定義
## 認証移行対応 + /signin 通知UI + オンボーディング + 初回AIチュートリアル

---

## 1. 背景

探Qメイトでは、従来の脆弱な認証方式から、Supabase Auth を標準とする認証方式へ移行した。  
この変更により、既存の α版ユーザーも、新しいアカウントとして再登録が必要になる。

一方で、過去の対話データの引き継ぎを必要とするユーザーは一部に限られる想定である。  
そのため、全ユーザー一律のデータ移行は行わず、**希望者のみ手動で引き継ぎ対応**を行う。

また、この認証移行は単なる登録し直しではなく、**初回体験全体を整える機会**として位置づける。  
本要件定義では、以下を一体で設計する。

- `/signin` に掲載する「認証方法変更と再登録案内」の通知UI
- 認証移行後の再登録導線
- シンプルなオンボーディング
- 探Qメイトの価値を最初のAI対話で体験できる初回AIチュートリアル
- 最初の AI 対話までの体験設計

---

## 2. 目的

### 2.1 プロダクト目的
- ユーザーが「何をすればいいか」で迷わず利用開始できること
- 探Qメイトの価値を短く理解できること
- 初回利用時に、探究テーマ設定または相談内容入力から AI との最初の対話まで到達できること

### 2.2 認証移行対応としての目的
- 認証方法変更による混乱を防ぐこと
- 既存 α版ユーザーに、再登録が必要であることを明確に伝えること
- 過去データ引き継ぎ希望者だけを拾えること
- 引き継ぎ依頼を管理可能な状態にすること

---

## 3. 基本方針

### 3.0 現行実装対象: 4ステップオンボーディングと興味タグ

現行の `/onboarding` は、PCでも縦長のスマホ風操作面で進める以下の4ステップで実装する。太いプログレスバーは使わず、`次へ（あと2ステップ）` のようにボタン文言と小さなステップ表示で残り量を伝える。

1. **再登録・引き継ぎ確認**
   - 旧データ引き継ぎ希望の有無
   - 希望する場合のみ旧ユーザー名（必須）と補足（任意）
2. **基本情報**
   - 表示名（必須）
   - 学校コード（任意）
   - 学校コード確認後の学年・クラス・出席番号
3. **興味の選択**
   - 興味タグ（1つ以上必須）
4. **テーマ入力**
   - 探究テーマ（任意）
   - 問い（任意）
   - 仮説（任意）
   - `あとで決める` で空のまま完了できる

興味タグは、候補ボタンの複数選択と自由入力で追加できる。入力値は空白を除去し、空文字・重複・長すぎるタグを保存しない。

保存先は `profiles.interests text[] not null default '{}'` とする。`profiles.interests` は生徒本人の自己申告による好きなもの・気になるもののタグであり、AI支援の入口として使う。先生向けに raw data としてそのまま公開する情報ではない。

既存ユーザーは `name` が設定済みであればオンボーディング完了扱いを維持し、興味未設定を理由に再オンボーディングへ強制遷移させない。

### 3.0.1 現行優先: 学校利用向け初回AIチュートリアル

当面は学校利用を中心に開始するため、登録前のゲストAI体験は実装しない。初回導線は次を標準とする。

```text
LP
  ↓
/signup または /signin
  ↓
/onboarding
  ↓
/chat?tutorial=first-ai
  ↓
通常利用
```

生徒は、オンボーディング完了後に初回AIチュートリアルへ進む。`profiles.first_ai_tutorial_completed = true` になるまでは、会話履歴、新規チャット、日誌、プロフィールなどの他操作へ移動できない。

初回AIチュートリアルは、オンボーディングで入力した興味タグ、探究テーマ、問い、仮説から初回相談文を下書きし、生徒が送信する。初回AI応答後に会話スタイルを「考えを深めたい」へ切り替え、2回目の送信とAI応答またはエラー表示を確認したあと、授業の最後に使う「今日の探究を記録する」導線を案内して完了する。スキップや閉じる操作による完了はできない。

### 3.1 初回体験の構成
初回体験は、以下の2段階で構成する。

1. **オンボーディング**
   - 引き継ぎ有無の確認
   - 最低限のプロフィール設定
2. **学校利用向け初回AIチュートリアル**
   - オンボーディング内容から初回相談文を下書きする
   - 生徒がAIへ最初の相談文を送信する
   - 初回AI応答後に「考えを深めたい」へ切り替える
   - 2回目のAI応答またはエラー表示後に日誌導線を案内し、通常利用へ進む

### 3.2 設計方針
- ユーザー種別の専用分岐画面は作らない
- 1本の標準オンボーディングフローの中で必要な情報だけを聞く
- 過去データ引き継ぎは「はい / いいえ」で選ばせる
- 「はい」を選んだユーザーのみ、旧ユーザー名を入力して申請する
- データ引き継ぎそのものは自動化せず、開発者が手動で対応する
- 初回AIチュートリアルは、説明を読むだけで終わらせず、実際の AI 対話まで導く
- `/signin` には、認証変更を確実に伝える通知UIを常設する

## 7. ルーティングと状態遷移

### 7.1 ルーティング制御ルール

* すべての protected route から `/signin` に遷移

* `/onboarding` に遷移

#### オンボーディング完了かつ初回AIチュートリアル未完了

* `/chat?tutorial=first-ai` に遷移

#### 両方完了

* 通常画面へ遷移

---

### 7.2 状態定義

#### ユーザー状態

* `onboarding_completed: boolean`
* `first_ai_tutorial_completed: boolean`
* `first_ai_tutorial_completed_at: string | null`

#### 任意で持つ補助状態

* `onboarding_step: string | null`

---

### 7.3 状態遷移

```text

  ↓
認証成功
  ↓
profiles 取得
  ↓
onboarding_completed = false
  → /onboarding

onboarding_completed = true かつ first_ai_tutorial_completed = false
  → /chat?tutorial=first-ai

onboarding_completed = true かつ first_ai_tutorial_completed = true
  → 通常利用

## 8. 画面要件

# 8.1 `/signin`

### 目的

* ユーザーをログインさせる
* 認証方法変更と再登録の必要性を明示する
* 既存ユーザーの混乱を減らす
* 新規登録導線へ自然につなぐ

### 必須要素

* ログインフォーム
* 認証方法変更に関する通知UI
* 新規登録への導線
* 必要ならパスワード再設定導線
* 必要なら FAQ / ヘルプ導線

### 成功時

* セッション成立後、状態に応じて `/onboarding` または次画面へ遷移

---

## 8.1.1 `/signin` 通知UI（認証方法変更・再登録案内）

### 目的

* 認証方式の変更を明確に伝える
* 既存の α版ユーザーが「ログインできないのは不具合か仕様か」で迷わないようにする
* 再登録が必要であることを明確に案内する
* 過去データ引き継ぎは、再登録後に希望者のみ申請できることを伝える

### UIの役割

`/signin` ページに表示する通知UIは、単なる補足文ではなく、**ログインフォームの前提条件を説明するための重要な notice / alert UI** とする。

ログインフォームの上部に常設し、ユーザーがログイン操作を行う前に必ず視認できる位置に表示する。

### 表示対象

* `/signin` にアクセスしたすべてのユーザー

### 表示条件

* 認証移行期間中は常時表示
* 将来的に feature flag で表示/非表示を切り替え可能にする

#### 例

* `VITE_SHOW_AUTH_MIGRATION_NOTICE=true`

### UI種別

以下のいずれかで実装する。

* Alert
* Notice Banner
* Notice Card

視認性と情報のまとまりを優先し、通常本文より強調された見た目にする。

### 表示位置

* `/signin` ページ内のログインフォームより上
* ページタイトルまたはサブタイトルの下
* スクロールしなくても見える範囲を優先する

### 配置優先順位

1. ページタイトル
2. 通知UI
3. ログインフォーム
4. 補助リンク（新規登録、パスワード再設定など）

### 通知UIに含める内容

最低限、以下を含める。

1. 認証方式が変更されたこと
2. 以前の α版ユーザーも再登録が必要なこと
3. 以前の認証情報ではそのままログインできないこと
4. 過去データ引き継ぎは、再登録後に希望者のみ申請できること
5. 新規登録ページへの導線

### 推奨見出し

* 認証方法の変更のお知らせ

### 推奨本文

> 探Qメイトでは、セキュリティ向上のため認証方法を変更しました。
> この変更に伴い、これまでの α版をご利用いただいていた方も、再度アカウント登録が必要です。
> 以前の認証情報ではそのままログインできません。
> 過去の対話データを引き継ぎたい場合は、再登録後に申請できます。

### CTA要件

#### 主導線

* `新規登録する`

### CTA遷移先

* `新規登録する` → `/signup`

### 見た目要件

* 通常の補足テキストより明確に目立つこと
* ただし、エラー表示と誤認されないこと
* 警告色に寄せすぎず、「重要なお知らせ」として見えること
* モバイルでも読みやすい改行・余白を確保すること

### 視覚設計の方向性

* information / notice 系アイコンを付与可能
* 背景色・枠線・余白によりカードとして認識できること
* 行間を十分に取り、長文に見えすぎないこと
* CTAボタンは本文の直下または右下に配置する

### インタラクション要件

* 初期実装では閉じるボタンは不要
* 移行期間中は常設表示とする
* ログイン失敗時にも通知UIは消えない

### エラーメッセージとの関係

ログイン失敗時は、通知UIとは別に通常のログインエラーを表示する。
ただし、通知UIの内容と矛盾しないこと。

#### 補助文言例

> ログインできない場合は、ページ上部の「認証方法の変更のお知らせ」もご確認ください。

### 文言設計上の注意

* 旧認証の脆弱性や内部事情を必要以上に強く表現しない
* ユーザー向けには「セキュリティ向上のため変更した」という表現に留める
* 不安を煽る表現は避ける
* 「以前のアカウントは使えない」ではなく、「再登録が必要」という行動ベースで伝える

### アクセシビリティ要件

* 通知UIは視覚的に本文と区別できること
* スクリーンリーダーで意味が伝わる構造にすること
* CTAはキーボード操作可能であること
* 明確なボタンまたはリンクを用意すること

### レスポンシブ要件

* モバイル幅では縦積みレイアウトに崩れること
* CTAボタンが本文と重ならないこと
* 画面幅が狭くても通知全体が読めること
* 長文が極端に詰まらないこと

# 8.2 `/onboarding`

### 目的

* 過去データ引き継ぎの意思確認
* 最低限のプロフィール設定
* 初回AIチュートリアルへの接続

### 画面構成

以下の4ステップを1本のフローとして実装する。

1. 引き継ぎ有無確認
2. プロフィール設定
3. 興味の選択
4. テーマ入力

PC画面でも主操作は縦長のスマホ風パネル内に集約する。プログレスバーは表示せず、`1/4` の小さな位置表示と `次へ（あと1ステップ）` のようなボタン文言で終わりまでの距離を伝える。

---

## 8.2.1 Step 1: 引き継ぎ有無確認

### 目的

* 旧データ引き継ぎ希望者のみを抽出する

### 質問文

* 以前の対話データを引き継ぎますか？

### 選択肢

* はい
* いいえ

### 「はい」の場合に表示する入力項目

* 旧ユーザー名（必須）
* 備考（任意）

### 備考の用途

以下のような補足を任意で入力できるようにする。

* 以前使っていた表示名
* 学校名
* 利用していた時期
* その他照合の手がかり

### 保存仕様

「はい」の場合、`migration_requests` に申請レコードを作成する。

### バリデーション

* 「はい」の場合、旧ユーザー名は必須
* 「いいえ」の場合、追加項目は不要

### UX要件

* 説明は短くする
* 「希望する方のみ対応します」と明示する
* ここで詳細説明をしすぎない

---

## 8.2.2 Step 2: プロフィール設定

### 目的

* 利用開始に必要な最低限のプロフィールを設定する

### 入力項目

* 表示名（必須）
* 学校コード（任意）
* 学年（学校コード入力時のみ表示）
* クラス（学校コード入力時のみ表示）

### UI仕様

* 初期状態では「表示名」「学校コード」のみ表示
* 学校コード入力時に「学年」「クラス」を表示
* 学校コード未入力でもそのまま次へ進める

### バリデーション

* 表示名は必須
* 学校コードは任意
* 学校コードが入力されている場合、学年・クラスは任意または必須を別途判断

  * 初期実装では任意でも可
  * 学校運用上必要なら後から必須化可能な設計にする

### 保存先

* `profiles`

---

## 8.2.3 Step 3: 興味の選択

### 目的

* 生徒の好きなもの・気になるものをAI支援の入口として受け取る
* テーマ未確定でも次のステップへ進める

### 入力項目

* 興味タグ（1つ以上必須）
* 候補タグの複数選択
* 自由入力によるタグ追加

### UX要件

* 単語レベルでよいことを明示する
* 後からプロフィールで更新できることを伝える
* 興味未選択では次へ進めない

---

## 8.2.4 Step 4: テーマ入力

### 目的

* すでに探究テーマや問いがある生徒だけ入力できるようにする
* テーマが決まっていない生徒を止めず、興味から初回AIチュートリアルへ進ませる

### 入力項目

* 探究テーマ（任意）
* 問い（任意）
* 仮説（任意）

### UI仕様

* `あとで決める` / `スキップして始める` 相当の導線を用意する
* 空のままでもオンボーディングを完了できる
* 入力した場合のみ `profiles.theme` / `profiles.question` / `profiles.hypothesis` に保存する

### 完了時処理

* 現行実装では `profiles.name` が保存されるとオンボーディング完了扱いになる
* 引き継ぎ希望時は `migration_requests` に申請を保存する
* 生徒は `/chat?tutorial=first-ai` へ遷移
* 先生・管理者はロール別の通常画面へ遷移

---

# 8.3 `/chat?tutorial=first-ai`

### 目的

* 生徒が初回からAIとの対話体験に到達できるようにする
* オンボーディングで入力した内容を、最初の相談文として自然に使う
* 学校利用中に迷子にならないよう、完了まで他操作を制限する

### 表示対象

* `role = "student"`
* オンボーディング完了済み
* `profiles.first_ai_tutorial_completed = false`

先生・管理者は対象外とし、ロール別の通常画面へ遷移する。

### 初回相談文

オンボーディング情報から、入力欄に初回相談文を下書きする。
チャット本文には、通常の `/chat` と同じAI側の初期メッセージを必ず表示する。チュートリアル案内カードは画面上部ではなく、入力欄の直上に表示し、その時点で必要な1操作だけを短く案内する。

テーマがある場合:

```text
「{theme}」について、今の問いや仮説をもとに次に何を調べるとよいか一緒に整理したいです。
```

テーマがない場合:

```text
興味がある「{interests}」から探究テーマを考えたいです。最初の問いを一緒に整理したいです。
```

### 操作制限

完了状態がDBに保存されるまでは、以下を利用できない。

* 別ルートへの直接移動
* 新規チャット
* 会話履歴
* 日誌
* プロフィール・アカウント設定
* チュートリアルのスキップ、閉じる、任意終了

### 完了条件

次の順序を満たした後、自動で完了する。

1. 初回相談文を送信する
2. 初回AI応答またはエラー表示を確認する
3. 会話スタイルを「考えを深めたい」に切り替える
4. 2回目のメッセージを送信する
5. 2回目のAI応答またはエラー表示を確認する
6. 右上の「今日の探究を記録する」導線を案内する

確認ボタンなどの手動完了操作は表示しない。AI失敗は完了可能とし、授業中に詰まらないようにする。

日誌導線の案内では、画面右上の「今日の探究を記録する」ボタンに注意が向くよう、ボタン以外を薄くグレーアウトし、吹き出しUIで「授業の最後にやってみる」機能として紹介する。この場で日誌作成フローは開始しない。

### 完了時処理

* `POST /auth/first-ai-tutorial/complete` を呼び出す
* `profiles.first_ai_tutorial_completed = true`
* `profiles.first_ai_tutorial_completed_at = now()`
* プロフィールを再取得する
* 現在表示中の会話を保持したまま、通常の `/chat` へ戻す
* 完了時に会話履歴の再読み込みで画面上の会話を置き換えない

## 9. データ設計

# 9.1 `profiles`

### 目的

* アプリ側プロフィール情報を管理する
* オンボーディングと初回利用の状態を管理する

### 必須カラム案

```sql
id uuid primary key
name text null
role text not null default 'student'
school_id uuid null
school_code_locked boolean not null default false
grade text null
class_name text null
attendance_number integer null
interests text[] not null default '{}'
theme text null
question text null
hypothesis text null
first_ai_tutorial_completed boolean not null default false
first_ai_tutorial_completed_at timestamptz null
created_at timestamptz not null default now()
updated_at timestamptz not null default now()
```

### 補足

* `id` は `auth.users.id` を参照する
* 現行実装では `name` が設定済みであればオンボーディング完了扱いとする
* 初回AIチュートリアルの完了状態は `profiles` を正とし、localStorage では管理しない
* `first_ai_tutorial_completed_at` は初回完了時刻を保持する

# 9.2 `migration_requests`

### 目的

* 過去データ引き継ぎ希望者の申請を保存する

### 必須カラム案

```sql
id bigserial primary key
new_user_id uuid not null
new_user_email text null
legacy_username text not null
note text null
status text not null default 'pending'
created_at timestamptz not null default now()
updated_at timestamptz not null default now()
```

### `status` 値

* `pending`
* `completed`
* `rejected`

### 補足

* `new_user_id` は `profiles.id` または `auth.users.id` に紐づく
* 初期実装では `pending` のみ運用できれば十分
* 後から管理画面対応しやすい構造にする

## 13. 受け入れ条件

# 13.1 `/signin`

* `/signin` に認証変更通知UIが表示される
* 通知UIはログインフォームより上に配置される
* 通知UI内に「認証方法の変更」「再登録が必要」「引き継ぎは再登録後に申請可能」の内容が含まれる
* 通知UIから `/signup` へ遷移できる
* ログイン失敗時にも通知UIが消えない
* モバイル・PC の両方で読みやすく表示される
* feature flag で表示制御可能である

# 13.2 オンボーディング

* 認証後、未完了ユーザーは必ず `/onboarding` に遷移する
* 引き継ぎ有無の選択ができる
* 引き継ぎ「はい」の場合のみ旧ユーザー名入力欄が表示される
* 表示名を保存できる
* 学校コード入力時のみ学年・クラス欄が表示される
* 興味選択とテーマ入力が別ステップで表示される
* 興味タグは1つ以上選択しないと進めない
* テーマ、問い、仮説は未入力でもスキップして完了できる
* 太いプログレスバーではなく、残りステップがボタン文言や小さな表示で伝わる
* PCでも縦長のスマホ風操作面として表示される
* 完了後に `onboarding_completed = true` になる
* 生徒は完了後に `/chat?tutorial=first-ai` へ遷移する
* 先生・管理者は強制チュートリアル対象外としてロール別通常画面へ遷移する

# 13.3 移行申請

* 引き継ぎ希望時に `migration_requests` が作成される
* 申請内容に新ユーザーID・旧ユーザー名・備考が保存される
* 同一ユーザーが完了後も通常利用できる

# 13.4 初回AIチュートリアル

* 新規生徒はオンボーディング後に `/chat?tutorial=first-ai` へ遷移する
* 未完了生徒は `/diary` `/profile` などへ直接移動できない
* 初回相談文がプロフィールの `theme` または `interests` から生成される
* 通常の `/chat` と同じAI側の初期メッセージがチャット本文に表示される
* チュートリアル案内カードは入力欄の直上に表示され、画面上部固定では表示されない
* 初回AI応答後に「考えを深めたい」の選択案内が表示される
* 2回目のAI応答またはエラー表示後に、右上の「今日の探究を記録する」導線が案内される
* 確認ボタンなどの手動完了操作なしで完了APIが呼び出される
* 完了API呼び出し後、プロフィール再取得を行い、現在の会話を保持したまま通常の `/chat` へ戻る
* teacher/admin は強制チュートリアル対象外である

## 15. 実装上の注意

### 15.1 分岐は増やしすぎない

* 「旧ユーザーですか？」の専用画面は作らない
* 必要なことだけ聞く

### 15.2 初回体験を長くしすぎない

* 説明のための説明画面を増やさない
* 入力と対話を早めに始めさせる

### 15.3 テーマ未確定ユーザーを想定する

* テーマを持っていないと進めない設計にしない
* 興味関心や悩みから入れるようにする

### 15.4 移行申請は申請だけに留める

* オンボーディング中にデータ移行処理まではしない
* 手動対応しやすい情報を残すことを優先する

### 15.5 DBを正とする

* 完了状態をフロントローカルだけで持たない
* リロードや別端末でも状態が壊れないようにする

### 15.6 `/signin` 通知UIは閉じられない前提でよい

* 移行期間中は見失わせないことを優先する
* dismiss できる仕様は後回しでよい

## 16. 最終仕様まとめ

今回の初回体験は、以下を標準フローとする。

1. ユーザーは新しい認証方式で登録 / ログインする
2. `/signin` では、認証方法変更と再登録案内の通知UIを表示する
3. `/onboarding` で

   * 過去データ引き継ぎの有無
   * 表示名
   * 学校コードと任意の追加情報
   * 興味タグ、探究テーマ、問い、仮説
     を設定する
4. 生徒は `/chat?tutorial=first-ai` で

   * オンボーディング内容から作られた初回相談文を確認する
   * AIチャットで最初の対話を始める
   * 初回AI応答またはエラー表示を確認する
   * 会話スタイルを「考えを深めたい」に切り替える
   * 2回目の対話後に「今日の探究を記録する」導線を確認する
5. `POST /auth/first-ai-tutorial/complete` により `profiles.first_ai_tutorial_completed = true` になった時点で初回AIチュートリアル完了とする
