# 探Qメイト「今日の探究を記録する」機能 改善仕様書

- 作成日: 2026-05-24
- 対象: 探Qメイト / 探究日誌生成機能
- ステータス: 実装用ドラフト
- 前提: 既存の改善前画面を置き換える。既存コードのうち、感情選択UIは流用し、🔥スライダーUIは通常導線から外したうえでA/Bテスト用に残す。

---

## 1. 目的

本機能は、探究学習における生徒の「行動の記録」ではなく、今日の探究を通じて生まれた感情、違和感、好奇心、迷い、納得感などの「心の状態」を、生徒自身の言葉で記録するための機能である。

改善後の体験では、AIが日誌本文を完成させるのではなく、AIは生徒が自分の状態を見つめ直すための「見立て」を提示する。最終的に保存・先生に共有される中心情報は、生徒自身が最後に入力した記録文とする。

---

## 2. 改善方針の要約

### 2.1 変更すること

| 項目 | 現状 | 改善後 |
|---|---|---|
| 起動導線 | チャット入力欄付近のキラキラボタン | 画面右上に常時表示する「今日の探究を記録する」ボタン |
| キラキラボタン | 日誌起動またはAI補助のように見える | 削除 |
| 感情選択 | 複数選択後に「一番近い気持ち」を選ぶ | 複数選択のみ。確認UIや一番近い気持ち選択は削除 |
| 🔥スライダー | 通常フローに含まれる | 通常フローから削除。A/Bテスト用にコードは残す |
| AIの役割 | AI視点の日誌を作り、編集する | AIは見立てを提示するだけ。編集対象にはしない |
| 最終入力 | AI文を編集する体験 | AI見立てを読んだあと、生徒が自分の言葉で記録する |
| 最終画面 | 編集フォーム / 共有内容が曖昧 | 先生用ダッシュボードに表示される内容のプレビュー |
| 先生共有 | AI見立てやsummaryの扱いが曖昧 | AI見立て全文は先生に見せない。生徒の記録文を中心に共有 |

### 2.2 確定した仕様判断

- AI見立て全文は先生用ダッシュボードに表示しない。
- 感情選択は複数選択とする。
- 感情選択後は「次へ」ボタンで手動進行する。
- 「一番近い気持ち」を選ばせる二重選択は廃止する。
- 🔥スライダーステップは通常フローから削除する。
- 🔥スライダーの実装コードは削除せず、feature flagで再検証可能にする。
- 最後の画面は編集画面ではなく、先生用ダッシュボード表示のプレビュー画面とする。

---

## 3. 体験設計の原則

1. **主語は生徒に置く**  
   AIが日誌を書くのではなく、生徒が自分の状態を記録する。

2. **AIは補助者として扱う**  
   AIは評価・診断・判定を行わず、生徒が自分の感覚を見直すための見立てを提示する。

3. **先生に見える内容を明確にする**  
   送信前に、先生用ダッシュボードに表示される内容を必ずプレビューする。

4. **心の状態を評価しない**  
   熱量の高さ、ポジティブさ、進捗の有無を良し悪しとして扱わない。

5. **画面ごとの目的を1つに絞る**  
   感情を選ぶ、AI見立てを見る、自分の記録を書く、先生表示を確認する、のように画面目的を分離する。

---

## 4. 新しいユーザーフロー

```text
チャット画面 / 任意の主要画面
  ↓
画面右上「今日の探究を記録する」
  ↓
Step 1. 感情を選ぶ
  ↓ 次へ
Step 2. AI見立て生成中
  ↓ 生成完了
Step 3. AIの見立てを見る
  ↓ 自分の記録を書く
Step 4. 自分の言葉で今日の探究を記録する
  ↓ プレビューを見る
Step 5. 先生に表示される内容を確認する
  ↓ 保存して先生に共有する
完了
```

UI上のステップ表示は、ローディングを独立ステップとして見せず、次の4ステップにする。

```text
1. 気持ちを選ぶ
2. 探Qメイトの見立てを見る
3. 自分の記録を書く
4. 先生への表示を確認する
```

## 5. 起動導線仕様

### 5.1 配置

画面右上に、日誌起動用のボタンを常時表示する。

```text
今日の探究を記録する
```

### 5.2 表示場所

- デスクトップ: 画面右上のヘッダー領域。
- モバイル / 横幅が狭い画面: ヘッダー右側、または折りたたみメニュー内。ただしチャット入力欄内には置かない。
- チャット入力欄、送信ボタンの近くには日誌起動導線を置かない。

### 5.3 表示文言

| 状態 | 文言 |
|---|---|
| 今日未記録 | 今日の探究を記録する |
| 今日記録済み | 今日の記録を見る |
| 追記機能を実装する場合 | 今日の記録に追記する |

初期実装では「今日未記録」と「今日記録済み」の2状態を推奨する。

### 5.4 削除する導線

チャット入力欄付近にあるキラキラボタンは削除する。

削除理由:

- 送信ボタンやチャット補助操作と意味が混ざる。
- 日誌機能の入口として軽く見えすぎる。
- 誤タップの可能性がある。
- 「探究を記録する」という独立した体験に見えない。

---

## 6. Step 1: 感情を選ぶ

### 6.1 目的

今日の探究を通じて、今残っている心の動きに近いものを選ぶ。ここでは正確な自己分析を求めず、AI見立ての材料として扱う。

### 6.2 画面タイトル

```text
今の気持ちに近いものを選んでください
```

### 6.3 説明文

```text
ぴったりでなくて大丈夫です。
今日の探究で残っている気持ちに近いものを選んでください。
```

### 6.4 選択ルール

- 複数選択可能。
- 最大選択数は既存仕様を踏襲し、最大4つとする。
- 1つ以上選択すると「次へ」ボタンを有効化する。
- 「一番近い気持ち」を選ばせるUIは廃止する。
- 選択後の確認カード、選択済み詳細エリアは表示しない。
- 選択状態はカード自体の見た目で表現する。

### 6.5 感情カード

既存の感情カード群を原則そのまま利用する。

現行例:

```text
ワクワク
すっきり
モヤモヤ
不安
行き詰まり
おもしろい
むずかしい
たのしい
```

感情マスタの追加・変更は本仕様の必須範囲外とする。追加する場合は別チケットで扱う。

### 6.6 フッター

フッターには選択数と次へボタンのみを表示する。

```text
2 / 4 選択中

［次へ］
```

### 6.7 次へ押下時の処理

「次へ」を押した時点でAI見立て生成を開始する。

処理:

1. 選択した感情IDを保存する。
2. 今日の対話ログ、探究テーマ、選択感情をもとにAI見立て生成APIを呼び出す。
3. 画面をローディング状態へ遷移する。

---

## 7. 削除: 🔥スライダーステップ

### 7.1 通常フローからの削除

「探究の熱量を置く」ステップは通常の探究日誌フローから削除する。

削除理由:

- 熱量が高いほど良い、低いと悪いという評価に見えやすい。
- やる気測定やモチベーション評価として受け取られるリスクがある。
- 先生に見られる可能性がある場合、生徒が正直に低い状態を選びにくい。
- 炎の演出が強く、心の記録よりもUI操作が主役になりやすい。
- 感情選択と役割が重複する。

### 7.2 コードの扱い

実装コードは削除しない。感情選択の別アプローチとして検証できるように、feature flagで残す。

推奨feature flag:

```ts
const diaryReflectionVariant = "emotion_cards" | "heat_slider";
```

初期値:

```ts
diaryReflectionVariant = "emotion_cards";
```

### 7.3 heat_slider variantの制約

`heat_slider` variantは以下の条件でのみ表示可能とする。

- 開発環境
- 検証ユーザー
- 明示的なA/Bテスト対象ユーザー

通常の本番導線では表示しない。

---

## 8. Step 2: AI見立て生成中

### 8.1 目的

感情選択後、AIが今日の探究の流れや心の動きを整理していることを伝える。

### 8.2 ローディング文言

推奨文言:

```text
探Qメイトが、今日の探究の流れを整理しています
```

補足文:

```text
選んだ気持ちや、今日の対話をもとに、
今の探究がどんな状態に見えるかをまとめています。

これは正解ではありません。
次の画面で、自分の感じ方と比べてみてください。
```

### 8.3 「分析」という言葉の扱い

「AIが今日の探究のプロセスを分析しています」という表現は使用可能だが、評価・診断に見えるリスクがある。基本文言では「分析」よりも「整理」「見立て」を優先する。

### 8.4 長時間生成時のフォールバック

15秒以上AI応答が返らない場合、以下の導線を表示する。

```text
見立ての作成に時間がかかっています。
AIの見立てなしで、自分の記録を書くこともできます。

［このまま記録を書く］
```

この場合、`aiObservationStatus = "skipped"` として記録画面へ進める。

### 8.5 生成失敗時

AI生成に失敗した場合は、記録作成自体を止めない。

表示文言:

```text
探Qメイトの見立てを作れませんでした。
選んだ気持ちをもとに、自分の記録を書くことができます。
```

アクション:

```text
［自分の記録を書く］
```

---

## 9. Step 3: AIの見立てを見る

### 9.1 目的

AIの見立てを、生徒が自分の感覚と照らし合わせるための材料として表示する。

この画面ではAI見立てを編集させない。AI見立てに対する「合っている」「違う」などの選択も必須にしない。

### 9.2 画面タイトル

```text
探Qメイトから見えたこと
```

### 9.3 説明文

```text
これは正解ではありません。
違うと思ったところや、しっくりこないところも、
次の記録にそのまま書いてください。
```

### 9.4 表示形式

AI見立てはカード形式で表示する。最大3カードを推奨する。

推奨カード:

1. 今日の心の動き
2. 探究の流れ
3. 残っていそうな問い

表示例:

```text
今日の心の動き
モヤモヤと不安がありつつ、少し整理したい気持ちもありそうです。

探究の流れ
今日の対話では、答えを決めるよりも、問いが広がっている段階に見えます。

残っていそうな問い
〇〇について、まだ引っかかりが残っているかもしれません。
```

### 9.5 操作

主要ボタン:

```text
自分の記録を書く
```

戻るボタン:

```text
戻る
```

戻る場合は感情選択画面へ戻る。感情が変更された場合、AI見立ては再生成する。

### 9.6 この画面でやらないこと

- AI見立て本文を編集させない。
- AI見立てを日誌本文に自動挿入しない。
- AI見立てを先生共有対象として表示しない。
- 「合っている / 違う」などの評価操作を必須にしない。
- タグ、引用、summaryなどの追加入力欄を表示しない。

---

## 10. Step 4: 自分の言葉で記録する

### 10.1 目的

AIの見立てを読んだあと、生徒が自分自身の言葉で今日の探究を記録する。

この記録文が、先生用ダッシュボードに表示される中心情報となる。

### 10.2 画面タイトル

```text
自分の言葉で記録する
```

または:

```text
今日の探究を自分の言葉で残す
```

### 10.3 説明文

```text
探Qメイトの見立てを読んで、自分ではどう感じましたか。
合っていたこと、違ったこと、まだ言葉にできないことを、そのまま残してください。
```

### 10.4 入力欄ラベル

```text
今日の探究の記録
```

### 10.5 プレースホルダー

```text
例：
探Qメイトの見立てを見て、たしかにまだ答えを決めるより、
問いが広がっている感じがした。
でも、自分では少し焦りもある。
次は〇〇について、もう少し考えてみたい。
```

### 10.6 入力仕様

- テキストエリアは空の状態から開始する。
- AI見立て本文を初期値として挿入しない。
- 文字数上限は既存仕様に合わせて500字とする。
- 入力が1文字以上ある場合、「プレビューを見る」ボタンを有効化する。
- 空欄の場合は「プレビューを見る」ボタンを無効化する。

### 10.7 空欄時の補助

空欄時には、入力欄下に以下の補助文を表示する。

```text
一文だけでも大丈夫です。
まだ言葉にできない場合は、そのことをそのまま書いてください。
```

必要に応じて、定型文挿入ボタンを追加できる。

```text
［まだ言葉にできない、と記録する］
```

押下時に入力欄へ以下を挿入する。

```text
まだうまく言葉にできないが、今日の探究について考えが残っている。
```

### 10.8 先生共有の明示

入力欄の近くに、以下の説明を表示する。

```text
この記録は、先生用ダッシュボードに表示されます。
送信前に次の画面で確認できます。
```

---

## 11. Step 5: 先生表示プレビュー

### 11.1 目的

送信前に、先生用ダッシュボードに表示される内容を生徒が確認する。

この画面は編集画面ではない。内容を修正したい場合は、前の「自分の言葉で記録する」画面へ戻る。

### 11.2 画面タイトル

```text
先生に表示される内容を確認する
```

### 11.3 説明文

```text
この内容が先生用ダッシュボードに表示されます。
送信する前に、内容を確認してください。
```

### 11.4 プレビュー内容

先生用ダッシュボードに表示される内容のみを表示する。

表示項目:

| 項目 | 表示有無 | 備考 |
|---|---:|---|
| 日付 | 表示 | 記録日 |
| 探究テーマ | 表示 | 紐づく探究テーマがある場合 |
| 選択した感情 | 表示 | Step 1で選択した感情 |
| 生徒が書いた今日の探究の記録 | 表示 | Step 4の入力本文 |
| AI見立て全文 | 非表示 | 先生には見せない |
| 元のチャットログ | 非表示 | 先生には見せない |
| 入力途中の下書き | 非表示 | 先生には見せない |

### 11.5 プレビュー表示例

```text
先生用ダッシュボードでの表示

日付
2026年5月24日

探究テーマ
地域の水環境について

選んだ気持ち
モヤモヤ / 不安 / すっきり

今日の探究の記録
探Qメイトの見立てを見て、たしかにまだ答えを決めるより、
問いが広がっている感じがした。
でも、自分では少し焦りもある。
次は〇〇について、もう少し考えてみたい。
```

### 11.6 共有されない内容の明示

プレビューの下に、以下を表示する。

```text
先生に表示されないもの
・探Qメイトとの元の対話
・AIの見立て全文
・入力途中の下書き
```

### 11.7 操作

戻るボタン:

```text
戻って修正する
```

送信ボタン:

```text
保存して先生に共有する
```

送信ボタンを押すと、`teacherVisiblePayload` を保存し、先生用ダッシュボードに表示可能な状態にする。

## 12. 先生用ダッシュボード表示仕様

### 12.1 表示するもの

先生用ダッシュボードには、以下のみを表示する。

```text
・記録日
・生徒名
・探究テーマ
・選択した感情
・生徒が最後に入力した探究記録
・送信日時
```

### 12.2 表示しないもの

先生用ダッシュボードに以下は表示しない。

```text
・AI見立て全文
・AI見立てカード
・元のチャットログ
・入力途中の下書き
・AI生成時のプロンプト
・モデル名、modelVersionなどの内部情報
```

### 12.3 APIレベルの制約

先生用ダッシュボードAPIでは、先生に表示しない項目をレスポンスに含めない。

特に、以下は先生向けAPIレスポンスから除外する。

```ts
aiObservation
aiObservationCards
rawConversationLog
draftReflectionText
aiPrompt
modelVersion
```

## 13. データ設計

### 13.1 DiaryEntry

```ts
type DiaryEntryStatus =
  | "draft"
  | "submitted";

type AiObservationStatus =
  | "not_requested"
  | "generating"
  | "succeeded"
  | "failed"
  | "skipped";

type DiaryEntry = {
  id: string;
  studentId: string;
  inquiryId?: string;
  conversationId?: string;

  createdAt: string;
  updatedAt: string;
  submittedAt?: string;

  status: DiaryEntryStatus;

  selectedEmotionIds: string[];

  aiObservationStatus: AiObservationStatus;
  aiObservation?: DiaryAiObservation;

  studentReflectionText: string;

  teacherVisiblePayload?: TeacherVisibleDiaryPayload;
};
```

### 13.2 DiaryAiObservation

```ts
type DiaryAiObservation = {
  id: string;
  generatedAt: string;
  cards: DiaryAiObservationCard[];

  // 内部管理用。先生向けAPIには返さない。
  modelVersion?: string;
};

type DiaryAiObservationCard = {
  title: string;
  body: string;
};
```

### 13.3 TeacherVisibleDiaryPayload

```ts
type TeacherVisibleDiaryPayload = {
  diaryEntryId: string;
  studentId: string;
  inquiryId?: string;

  date: string;
  submittedAt: string;

  selectedEmotionIds: string[];
  studentReflectionText: string;
};
```

### 13.4 重要な分離

`aiObservation` と `teacherVisiblePayload` は分離する。

- `aiObservation`: 生徒のメタ認知を支援するための内部・生徒向け情報。
- `teacherVisiblePayload`: 先生用ダッシュボードに表示するための情報。

先生向け表示では `teacherVisiblePayload` のみを参照する。

## 14. API仕様案

実際の既存API構成に合わせてパスは調整してよい。ここでは実装イメージとして定義する。

### 14.1 今日の日誌状態取得

```http
GET /api/diary/today
```

レスポンス例:

```json
{
  "hasSubmittedToday": false,
  "diaryEntryId": null
}
```

用途:

- 右上ボタンの表示文言切り替え。
- 今日記録済みの場合の遷移先判定。

### 14.2 AI見立て生成

```http
POST /api/diary/ai-observation
```

リクエスト例:

```json
{
  "inquiryId": "inq_123",
  "conversationId": "conv_123",
  "selectedEmotionIds": ["moyamoya", "fuan"]
}
```

レスポンス例:

```json
{
  "aiObservationStatus": "succeeded",
  "aiObservation": {
    "id": "obs_123",
    "generatedAt": "2026-05-24T08:00:00.000Z",
    "cards": [
      {
        "title": "今日の心の動き",
        "body": "モヤモヤと不安がありつつ、少し整理したい気持ちもありそうです。"
      },
      {
        "title": "探究の流れ",
        "body": "今日の対話では、答えを決めるよりも、問いが広がっている段階に見えます。"
      },
      {
        "title": "残っていそうな問い",
        "body": "〇〇について、まだ引っかかりが残っているかもしれません。"
      }
    ]
  }
}
```

失敗時レスポンス例:

```json
{
  "aiObservationStatus": "failed",
  "aiObservation": null
}
```

### 14.3 日誌保存・先生共有

```http
POST /api/diary/entries
```

リクエスト例:

```json
{
```

## 15. AI見立て生成仕様

### 15.1 入力に使う情報

AI見立て生成では以下を利用する。

```text
・選択された感情
・今日の探Qメイトとの対話内容
・現在の探究テーマ
```

過去日誌の利用は初期実装では必須としない。将来的に利用する場合は、ユーザーに明示する。

### 15.2 出力形式

AIは日誌本文ではなく、構造化された見立てカードを返す。

```json
{
  "cards": [
    {
      "title": "今日の心の動き",
      "body": "..."
    },
    {
      "title": "探究の流れ",
      "body": "..."
    },
    {
      "title": "残っていそうな問い",
      "body": "..."
    }
  ]
}
```

### 15.3 文体ルール

AI出力は以下を守る。

- 断定しない。
- 評価しない。
- 診断しない。
- 生徒の感情を決めつけない。
- 「良い」「悪い」で表現しない。
- 「進んでいる / 遅れている」と評価しない。
- 先生目線の講評にしない。
- 生徒が自分の感覚と比べられる表現にする。

### 15.4 推奨表現

```text
〜のように見えます。
〜かもしれません。
〜という感じが残っている可能性があります。
違っていたら、その違和感も大事な記録になります。
```

### 15.5 禁止表現

```text
あなたは不安です。
あなたの課題は〇〇です。
今日は探究が進んでいません。
次は〇〇すべきです。
やる気が下がっています。
```

## 16. フロントエンド状態管理

### 16.1 Flow State

```ts
type DiaryFlowState =
  | "emotion_selecting"
  | "ai_generating"
  | "ai_observation_ready"
  | "student_reflection_editing"
  | "teacher_preview"
  | "submitting"
  | "submitted"
  | "error";
```

### 16.2 状態遷移

```text
emotion_selecting
  ↓ 次へ
ai_generating
  ↓ AI生成成功
ai_observation_ready
  ↓ 自分の記録を書く
student_reflection_editing
  ↓ プレビューを見る
teacher_preview
  ↓ 保存して先生に共有する
submitting
  ↓ 成功
submitted
```

失敗時:

```text
ai_generating
  ↓ AI生成失敗
student_reflection_editing
```

戻る操作:

```text
ai_observation_ready → emotion_selecting
student_reflection_editing → ai_observation_ready
teacher_preview → student_reflection_editing
```

感情選択に戻って選択内容が変わった場合、AI見立ては再生成する。

## 19. アクセシビリティ要件

- 感情カードはキーボード操作で選択可能にする。

- ボタンの無効状態には理由が伝わる補助文を置く。

- 先生に表示される内容と表示されない内容を視覚的にも構造的にも分ける。

## 26. 最終仕様まとめ

改善後の探究日誌機能は、画面右上の「今日の探究を記録する」から起動する。

生徒はまず、今日の探究で残っている気持ちを複数選択する。次へ進むと、探Qメイトが選択感情と今日の対話をもとに、探究の流れや心の動きを見立てる。

生徒はAIの見立てを読んだうえで、自分の言葉で今日の探究を記録する。AIの見立ては編集対象ではなく、記録のための材料として扱う。

最後に、先生用ダッシュボードに表示される内容をプレビューする。先生に表示されるのは、生徒が選んだ感情と、生徒が最後に入力した記録文を中心とする。AI見立て全文や元のチャットログは先生には表示しない。
