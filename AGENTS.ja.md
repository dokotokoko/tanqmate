# AGENTS.md

このドキュメントは、このリポジトリで作業する AI コーディングエージェントが最初に読むべき文書です。これは軽量な行動ガイドであり、機能仕様書、API リファレンス、デザインシステムではありません。

## 1. プロジェクト概要

探Qメイトは、高校生の探究学習を伴走するAIパートナーです。目的は、生徒に「どうしたら良いかわからない」を隣で同じ方向を向いて支え、答えを教えるではなく、歩む道を一緒に考えていくことです。
その過程の連続を通して、生徒の興味やテーマが発達していくことを支援します。

現在の実装ルート:

- `apps/frontend`: Vite + React + TypeScript フロントエンド
- `apps/backend`: FastAPI バックエンド
- `infra`: Docker Compose 設定
- `nginx`: ローカルリバースプロキシ
- `schema` / `apps/backend/schema`: Supabase および DB スキーマファイル

`backend/` や `react-app/` などの古いパスが、アーカイブ資料に登場することがあります。新規作業では必ず `apps/backend` と `apps/frontend` を使用してください。

## 2. 開発の基本原則

1. 答えの代替ではなく、探究を支援する。
2. 生徒の主体性を保つ。
3. UI は落ち着きがあり、読みやすく、評価的でない状態に保つ。
4. 生徒のプライバシーと学校関連データを保護する。
5. 小さく焦点を絞った変更を行う。
6. 新しいやり方を導入する前に、既存のプロジェクトパターンを優先する。
7. UI 実装は `DESIGN.md` に従う。
8. 機能の挙動は `docs/requirements/` に従う。
9. `archive` を現在の正本として使わない。

## 3. 正本ドキュメント

- `README.md`: 人間向けのプロジェクト概要とセットアップメモ
- `AGENTS.md`: AI エージェント向けの開発指示
- `DESIGN.md`: ビジュアルデザインシステムと UI トーン
- `docs/INDEX.md`: ドキュメントマップ
- `docs/workflows/DEVELOPMENT_WORKFLOW.md`: 開発ワークフローと仕様駆動開発
- `docs/requirements/`: プロダクト要件と機能要件
- `docs/architecture/`: システム、API、データモデルのドキュメント
- `docs/adr/`: 重要な意思決定記録
- `docs/checklists/`: 本番環境・リリースチェックリスト
- `docs/reviews/`: レビューレポート

`archive` は過去の成果物のみを保存する場所です。実装、デザイン、要件のために参照してはいけません。有用な内容がある場合は、まず現在のドキュメントに昇格させてから使用してください。

## 4. 必須ワークフロー

新機能、大きな変更、UI / UX フロー変更、API 変更、データベース変更、AI 挙動変更、プライバシー / セキュリティ関連の変更、または複数レイヤーにまたがる変更では、以下に従ってください。

`docs/workflows/DEVELOPMENT_WORKFLOW.md`

誤字修正、フォーマット変更、限定的なドキュメント編集、または挙動を変えない明らかなバグ修正では、軽量フローを使用してもかまいません。

## 5. 要件ワークフロー

機能の挙動を実装する前に、以下を行ってください。

1. `docs/requirements/README.md` を読む。
2. `docs/requirements/000-overview.md` を読む。
3. `docs/requirements/features/` 配下の関連ファイルを読む。
4. 変更がプライバシー、セキュリティ、アクセシビリティ、パフォーマンス、信頼性、AI 挙動に関わる場合は、`docs/requirements/010-non-functional.md` を読む。
5. 関連要件を満たす最小の変更を実装する。
6. 実装された挙動が現在の仕様を変える場合は、要件ドキュメントを更新する。

## 6. デザインシステムルール

`DESIGN.md` は、ビジュアルデザインの正本です。

UI を編集する場合:

- `DESIGN.md` で定義された色、タイポグラフィ、余白、角丸、コンポーネントパターンを使用する。
- `DESIGN.md` を更新しない限り、新しいビジュアルトークンを導入しない。
- 生徒向け UI は落ち着きがあり、明確で、支援的に保つ。
- 教員向け UI は一覧性を保ち、評価的なシグナルに注意する。
- AI 出力を最終回答のように見せない。
- 実装済みトークンには `apps/frontend/src/styles/design-system.ts` を使用する。

## 7. コーディングルール

- 既存のアーキテクチャ、命名、ディレクトリ境界を維持する。
- ユーザーの変更や無関係な作業を絶対に巻き戻さない。
- ルーターは薄く保ち、バックエンドのビジネスロジックは `apps/backend/services` に置く。
- FastAPI ルーターは `apps/backend/routers` に置く。
- フロントエンドのページは `apps/frontend/src/pages` に、再利用可能な UI は `apps/frontend/src/components` に置く。
- 新しい抽象化を追加する前に、既存の API クライアント、ストア、フック、共通コンポーネントを使用する。
- 既存ライブラリで合理的に解決できない場合のみ、依存関係を追加する。
- 原則OOPに従ってクリーンなコードベースを維持する
- コメントは最小限にする

## 8. AI / LLM 機能ルール

- AI は振り返り、問いかけ、計画、次の行動を支援するものであり、最終的な権威として振る舞ってはならない。
- AI 出力は提案または視点として表示し、生徒の主体性を保つ。
- 閲覧権限のないユーザーに、生の私的な振り返り、生のチャットログ、または機微な生徒文脈を公開しない。
- LLM 呼び出しは、既存のバックエンドクライアントとサービスを通す。
- 確立済みのプロンプト / サービス構造の外に、プロンプト、モデル挙動、API キーをハードコードしない。

## 9. プライバシーとセキュリティ

- シークレット、API キー、トークン、認証情報、`.env` ファイルを絶対にコミットしない。
- 明示的に必要で安全な場合を除き、生徒名、私的な振り返り、クラス情報、その他の個人データをログに出力しない。
- 権限のないユーザーに、生徒の私的な振り返りを公開しない。
- IP アドレス、ホスト名、環境固有 URL をハードコードしない。
- 認証、認可、CORS、データアクセスチェックを弱めない。
- 破壊的なデータマイグレーションや SQL を実行する前に確認する。

## 10. テストと品質確認コマンド

変更に関連するチェックを実行してください。チェックを実行できない場合は、その理由を報告してください。

フロントエンド:

```bash
cd apps/frontend
npm run build
```

バックエンド:

```bash
cd apps/backend
python -m pytest
```

Docker 開発環境:

```bash
docker compose -f infra/docker-compose.dev.yml up --build
```

## 11. ドキュメント更新ルール

挙動、セットアップ、API、スキーマ、UI ルール、AI 挙動、プライバシー / セキュリティ上の前提、またはリリースプロセスが変わる場合は、ドキュメントを更新してください。

適切な更新先を使用してください。

- 機能の挙動: `docs/requirements/`
- 技術アーキテクチャ: `docs/architecture/`
- ビジュアル / UI ルール: `DESIGN.md`
- 重要な意思決定: `docs/adr/`
- ワークフロールール: `docs/workflows/DEVELOPMENT_WORKFLOW.md`
- 本番環境チェック: `docs/checklists/`

## 12. 完了チェックリスト

完了を報告する前に、以下を確認してください。

- 変更が依頼範囲に収まっている。
- 現在の要件ドキュメントとデザインドキュメントに従っている。
- プライバシーと認可への影響を確認した。
- 関連するテスト / ビルドを実行した、または実行できなかった理由を明記した。
- 挙動またはプロセスが変わった場合は、ドキュメントを更新した。
- シークレット、生成物、無関係な変更を含めていない。

## 13. エージェントの振る舞い

- コードベースの仕組みを決めつける前に、コードベースを読む。
- 新しいアーキテクチャより、既存パターンを優先する。
- 最終報告は簡潔にし、変更ファイル、実行したチェック、残存リスクを含める。
- プロダクト、プライバシー、データマイグレーションに関する判断が曖昧で作業が止まる場合は、確認を求める。
